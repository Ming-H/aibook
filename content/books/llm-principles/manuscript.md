# 大语言模型原理

**从 Token 到 Transformer，从预训练到对齐——图解式干练讲解**

# Part 1: 基础构件

理解大语言模型的第一步，是搞清楚它的"原材料"——文字是怎么变成数字的，数字是怎么变成"理解"的。本部分将带你从 Token 到 Transformer，逐步拆解大模型的核心构件。

---

## §01 语言模型的本质——从猜下一个词开始
The Essence of Language Models: Predicting the Next Word

### 你说"今天天气"，它接"真好"——这不是魔法，是概率

打开 ChatGPT，输入"床前明月光"，它会毫不迟疑地接上"疑是地上霜"。你可能会觉得它"懂"这首诗。但事实上，它做的事情远比你想象的简单：**在所有可能的下一个词中，挑出概率最高的那个。**

这就是语言模型的全部工作原理——预测下一个词。

听起来太简单了？确实如此。但就是这个简单的想法，在被放大到数千亿参数、训练于数万亿 Token 的规模之后，产生了令所有人震惊的"智能涌现"。在拆解它的复杂构造之前，我们先从最根本的问题出发：为什么"猜下一个词"能产生智能？

### 语言模型 = 条件概率估计器

先建立一个精确的直觉：**语言模型本质上是一个条件概率估计器。**

给定一段文本的前 N 个词，语言模型计算第 N+1 个词是词典中每个候选词的概率，然后选择概率最高的那个（或者按概率随机采样）。

用日常语言描述：

```
输入："今天天气"
模型计算：P("真好" | "今天天气") = 0.35
         P("不好" | "今天天气") = 0.25
         P("怎么样" | "今天天气") = 0.15
         P("很冷" | "今天天气") = 0.12
         ...（其他所有词各分一点概率）

输出：选 "真好"（概率最高）
```

然后把这个预测出来的词加到已有文本末尾，再预测下一个词，再下一个……如此循环，直到生成一个完整的回答。

> [!NOTE]
> 严格来说，语言模型建模的是整个句子的联合概率 P(w₁, w₂, ..., wₙ)。根据概率论的链式法则，这个联合概率可以被分解为一系列条件概率的乘积：P(w₁) × P(w₂|w₁) × P(w₃|w₁,w₂) × ... × P(wₙ|w₁,...,wₙ₋₁)。所以"预测下一个词"不只是模型的输出方式，而是它对语言的全部理解。

### 从 N-gram 到 Transformer：语言模型 30 年简史

"预测下一个词"这个想法并不新。事实上，语言模型已经有超过 30 年的发展历史。让我们用一张时间线来理解这个演进。

想象一张这样的图：一条从左到右的时间线，上面标注着四个关键节点——

**1980 年代及更早：N-gram 语言模型**

N-gram 的思路极其朴素：统计历史文本中词的共现频率。比如要预测"今天天气"后面的词，就去看看历史上"今天天气"后面出现最多的词是什么。

```
统计语料库：
"今天天气真好" → 出现 500 次
"今天天气不好" → 出现 300 次
"今天天气很热" → 出现 150 次

结论："真好"概率最高 → 选它
```

N-gram 的致命问题是**上下文窗口太短**。Bi-gram（N=2）只看前 1 个词，Tri-gram（N=3）只看前 2 个词。N 越大，可能的组合数指数爆炸，统计所需的语料量也随之指数增长。这意味着 N-gram 根本无法捕捉长距离的语义依赖。

**2000 年代：RNN（循环神经网络）**

RNN 引入了一个核心改进：**隐状态（Hidden State）**。每处理一个词，RNN 会更新自己的隐状态，这个隐状态像是一个"记忆摘要"，把之前所有词的信息压缩进去。

想象一张这样的图：一个词条条从左到右经过同一个 RNN 单元，每一步产生一个隐状态，传给下一步。像一个传令兵，把每条信息依次传递下去。

RNN 解决了 N-gram 的窗口限制问题，但带来了新的麻烦——**梯度消失**。当句子太长时，传令兵走到句末，开头的信号已经衰减到几乎为零。也就是说，RNN 虽然理论上能记住无限长的上下文，实际上只能记住最近十几二十个词。

2014-2015 年，LSTM（Long Short-Term Memory，长短期记忆网络）和 GRU（Gated Recurrent Unit，门控循环单元）通过门控机制部分缓解了梯度消失问题，但本质瓶颈依旧存在：**串行处理，无法并行计算，长距离依赖仍然困难。**

**2017 年：Transformer —— "Attention Is All You Need"**

Google 的这篇论文标题本身就是一个宣言：注意力机制就是你需要的一切，不再需要循环。

Transformer 的核心创新是**自注意力机制**——让每个词直接和所有其他词建立联系，不管它们隔了多远。没有传令兵，没有信号衰减，每个词都能一步到位地"看到"所有其他词。

而且，因为不再需要像 RNN 那样一步步串行处理，Transformer 可以**完全并行化**——用 GPU 同时计算所有词之间的关系。这一特性让模型规模的大幅扩展成为可能。

**2018 年至今：GPT 系列——把简单想法做到极致**

OpenAI 的 GPT 系列走了一条被很多人初期低估的路线：只保留 Transformer 的 Decoder 部分，用"预测下一个词"这一个目标，疯狂增加模型规模和训练数据。

| 维度 | N-gram | RNN/LSTM | Transformer | GPT 系列 |
|------|--------|----------|-------------|----------|
| 上下文范围 | 固定 N 个词 | 理论无限，实际有限 | 完全全局 | 数万到数十万 Token |
| 长距离依赖 | 不支持 | 弱 | 强 | 强 |
| 并行计算 | 无需 | 不支持 | 完全并行 | 完全并行 |
| 参数规模 | 几乎为零 | 百万级 | 亿级 | 千亿到万亿级 |
| 涌现能力 | 无 | 无 | 初步 | 显著 |

### 为什么"预测下一个词"能产生智能涌现

这是最反直觉的部分。一个只会"猜下一个词"的模型，为什么能写代码、做推理、翻译语言、通过律师资格考试？

答案在于：**要极其准确地预测下一个词，你实际上需要理解整个世界。**

考虑这个例子：

```
输入："水在标准大气压下，温度降到 0°C 以下会"
```

要准确预测"结冰"这个词，模型必须"知道"物理规律。再看一个：

```
输入："小明把球放进杯子里，然后拿着杯子走出了房间，请问球现在在哪里？"
```

要预测出"球在杯子里，杯子在房间外面"这样的续写，模型必须具备空间推理能力。

**预测下一个词是一个看似简单但实际上要求极高的任务。** 当训练数据覆盖了互联网上几乎所有公开文本——科学论文、小说、代码、百科全书、论坛对话——要在这上面把预测做到极致，模型就不得不学会：

- **语法和语义**：否则无法生成语法正确的句子
- **事实知识**：否则无法准确续写关于真实世界的内容
- **逻辑推理**：否则无法预测论证过程的下一步
- **编程能力**：否则无法续写代码
- **多语言能力**：否则无法处理混合语言的文本

这就是"涌现"（Emergence）的含义：**简单的规则在足够大的规模上运行时，产生了规则设计者都没有预料到的高级能力。**

> [!TIP]
> 一个有用的类比：物理学中的相变。水分子之间的相互作用规则很简单——氢键吸引。但当 10²³ 量级的水分子聚集在一起时，就涌现出了"冰""液态水""水蒸气"这些截然不同的宏观状态。语言模型中的"智能涌现"也类似——下一个词预测的规则很简单，但千亿参数 + 万亿 Token 的规模下，涌现出了推理、规划、创作等高级能力。

### GPT 系列的演进：一个简单想法的极限放大

OpenAI 的 GPT 系列是"预测下一个词"这一范式最坚定的践行者。从 2018 年到 2023 年，GPT 系列用五年时间完成了一个想法从论文到产品的惊人跃迁。

**GPT-1（2018）：验证可能性**

1.17 亿参数，5GB 训练数据。核心贡献是验证了"用 Transformer Decoder 做语言模型预训练，然后在下游任务上微调"这条路线的可行性。它在 12 个 NLP 任务中的 9 个上达到了当时的最好水平。

**GPT-2（2019）：规模化初见成效**

15 亿参数，40GB 训练数据。GPT-2 的核心发现是：**不做微调，只用预训练的"预测下一个词"能力，就能在多种任务上表现不错。** 只要在输入中给出合适的"提示"，比如"将以下英文翻译为法语："，模型就能直接输出翻译结果。这就是"零样本学习"的雏形。

因为担心被滥用，OpenAI 一开始只发布了小版本模型——这段历史后来被称为"OpenAI 的保守时刻"。

**GPT-3（2020）：涌现开始显现**

1750 亿参数，570GB 训练数据。参数量从 15 亿跃升到 1750 亿，一个量级的跨越。GPT-3 展现出了令人惊讶的**上下文学习（In-Context Learning）**能力：你不需要修改模型的任何参数，只要在输入中给出几个例子，它就能理解你要做什么并执行。

```
输入：
将英文翻译为法文：
cheese → fromage
bread → pain
wine →

输出：vin
```

GPT-3 没有被专门训练来翻译，它只是看了几个例子就"理解"了任务。这种能力在小模型上几乎看不到，是典型的涌现现象。

**GPT-4（2023）：多模态 + 能力质变**

参数规模未公开（据推测在万亿级别），训练数据量也远超 GPT-3。GPT-4 在几乎所有 NLP 基准测试上都大幅超越 GPT-3，在模拟律师资格考试中得分进入前 10%，在生物奥林匹克竞赛中超越 99% 的考生。

GPT-4 的关键升级还包括：多模态能力（能看图）、大幅改进的推理能力、更长的上下文窗口、以及通过 RLHF（人类反馈强化学习）实现的对齐。

想象一张这样的图：一条向上的阶梯曲线，横轴是参数量（从 1 亿到万亿），纵轴是能力。前三个阶段（GPT-1/2/3）曲线平滑上升，到了 GPT-4，曲线出现了一个跳跃——这不仅仅是量变，而是质变。

> [!WARNING]
> 一个常见的误解是认为模型越大越好。实际上，GPT 系列的成功不仅来自参数规模的增长，还来自训练数据质量的提升、训练方法的改进（如 RLHF）、以及对齐技术（让模型的行为符合人类期望）的应用。规模是必要条件，但不是充分条件。

### 小结

语言模型的核心思想极其简单——预测下一个词。但这个简单思想背后隐藏着一个深刻的洞察：**要完美预测下一个词，你必须理解整个世界。** 从 N-gram 到 Transformer，技术路线不断演进，但目标始终如一。GPT 系列的成功证明了：当这个简单想法被推进到足够大的规模时，涌现出的能力超出了所有人的预期。

## §02 Token——模型眼中的文字世界
Tokens: How Models See Text

### 你看到"你好"，模型看到 [345, 9872]

当你在对话框里输入"你好，世界！"这六个字符时，你以为模型收到的是六个汉字。但实际上，在进入模型之前，这段文字已经被切分成了一串数字 ID：

```
"你好，世界！" → [345, 9872, 876, 4521, 2308]
```

这些数字 ID 就是 **Token**。Token 是模型处理文字的最小单元——模型从不直接操作字符或汉字，它的整个世界里只有 Token。

理解 Token 是理解大语言模型的第一步，因为它决定了模型能"看到"什么、能处理多长的文本、以及你使用 API 时账单上的数字。

### Token 是什么：介于字符和词语之间的单元

Token 既不等同于字符，也不等同于完整的词语。它是一种介于两者之间的**子词单元（Subword Unit）**。

用英文举例：

```
句子："unbelievable"
字符级切分：u, n, b, e, l, i, e, v, a, b, l, e  （12 个单元）
词语级切分：unbelievable                            （1 个单元）
Token 切分：un, believe, able                       （3 个单元）
```

用中文举例：

```
句子："人工智能"
字符级切分：人, 工, 智, 能                           （4 个单元）
词语级切分：人工智能                                  （1 个单元）
Token 切分：人, 工, 智能                             （3 个单元）
```

为什么要用这种"不上不下"的粒度？因为它在两个极端之间取得了最佳平衡：

- **字符级**的问题：粒度太细，序列太长，模型难以学到有意义的语义。"unbelievable" 被切成 12 个单元，每个字母单独看没有意义。
- **词语级**的问题：词表太大。中文有几十万个词，英文加上各种专有名词和缩写也有数十万。更大的问题是，总有新词不在词表中——模型遇到不认识的词就无法处理（OOV 问题）。
- **子词级（Token）的优势**：高频词保持完整，低频词被拆成更小的有意义的片段。任何文本都能被编码，永远不会遇到"不认识"的情况。

> [!NOTE]
> 一个经验性的规律：在英文中，1 个 Token 大约对应 4 个字符或 0.75 个单词。在中文中，1 个 Token 通常对应 1-2 个汉字。这意味着同样一段内容，中文消耗的 Token 数通常比英文多。这也是为什么使用中文与大模型对话时，成本往往更高。

### BPE 算法详解：从字节到子词的合并过程

目前最主流的 Tokenization 方法是 **BPE（Byte Pair Encoding，字节对编码）**。GPT-2/3/4、LLaMA、Claude 等主流模型都使用 BPE 或其变体。让我们用一个完整的例子来理解它的工作原理。

**训练阶段：构建词表**

BPE 的训练过程就是一个不断"合并高频相邻符号"的过程。
![图2-1：BPE分词算法流程——从字符到子词的迭代合并](images/bpe-tokenization.png)


初始状态，每个字符（或字节）都是独立的 Token。假设我们有以下训练语料：

```
low   lower   newest   widest   low
```

**第一步：统计所有相邻符号对的频率**

```
lo: 3    ow: 3    er: 2    ne: 1    ew: 2
we: 2    es: 2    st: 2    id: 1    de: 1
es: 2    t: 2     ...
```

**第二步：合并频率最高的符号对**

"lo" 和 "ow" 频率最高，合并它们：

```
low → 作为一个 Token
语料变为：[low] [low] er   newest   widest
```

**第三步：重复上述过程**

继续统计、继续合并，每次合并都产生一个新的 Token，直到词表达到预设大小。

```
合并顺序示例：
lo + ow → low
er → er（本身频率够高，作为独立 Token）
new + est → newest
...
```

最终，词表中有几万到十几万个 Token。其中既包括完整的高频词（如 "the"、"is"），也包括常用的子词片段（如 "un-"、"-ing"、"-tion"），还有单个字符和字节。

**推理阶段：用词表切分新文本**

当用户输入一段新文本时，Tokenizer 按照训练时学到的合并规则，从左到右将文本切分成 Token，然后查表将每个 Token 映射为词表中的整数 ID。

想象一张这样的图：一段文字 "I love machine learning" 从左端进入一个"切割机"，切割机根据 BPE 规则将它切成 ["I", " love", " machine", " learning"] 四个片段，每个片段查表变成数字 [40, 3021, 8765, 2944]，最终输出一串整数 ID 送入模型。

```
文字输入："I love machine learning"
     ↓ BPE Tokenizer
Token 序列：["I", " love", " machine", " learning"]
     ↓ 查词表
Token ID：  [40, 3021, 8765, 2944]
     ↓ 送入模型
模型处理...
```

> [!TIP]
> BPE 的一个重要特性是**贪婪且确定性的**：同样的文本，用同一个 Tokenizer，每次都会得到完全相同的 Token 序列。这意味着 Tokenization 是一个确定性的预处理步骤，不存在随机性。

### 不同 Tokenizer 有什么不同

虽然主流模型都用 BPE 或其变体，但具体实现存在差异，导致同样的文本在不同模型中被切分成不同的 Token。

| 模型 | Tokenizer 类型 | 词表大小 | 特点 |
|------|---------------|---------|------|
| GPT-4 | BPE (tiktoken) | ~100K | 多语言支持好，英文效率高 |
| Claude | BPE 变体 | ~100K | 对代码和多语言优化 |
| LLaMA | SentencePiece (BPE) | ~32K | 词表较小，中文效率偏低 |
| Gemini | SentencePiece | ~256K | 超大词表，多语言覆盖广 |

让我们看一个具体的对比。输入同样的文本："Hello, 你好世界！"

```
GPT-4 Tokenizer:
Token: ["Hello", ",", " 你", "好", "世界", "！"]
数量：6 个 Token

LLaMA Tokenizer:
Token: ["Hello", ",", " ▁", "你", "好", "世", "界", "！"]
数量：8 个 Token （中文被切得更碎）

Gemini Tokenizer:
Token: ["Hello", ",", " 你好", "世界", "！"]
数量：5 个 Token （大词表的优势）
```

词表大小直接影响模型处理多语言文本的效率。词表大的模型（如 Gemini）可以将常见的外语词组保持为完整 Token，而词表小的模型（如 LLaMA）则不得不把非英语文本切成更碎的片段，消耗更多 Token。

> [!WARNING]
> Tokenizer 的选择不仅影响"效率"，还影响模型的能力。研究表明，Tokenizer 的粒度会影响模型处理特定任务的效果。例如，在做算术运算时，如果数字被切成不合理的片段（如 "1234" 被切成 "12" 和 "34"），模型的计算准确率会明显下降。这也是为什么 GPT-4 的 Tokenizer 对数字做了专门优化，尽量避免在数字中间切分。

### 上下文窗口：为什么 GPT-4 能处理 128K Token

每个模型都有一个**上下文窗口（Context Window）**的限制——它能一次处理的 Token 数量的上限。这个限制不是 Tokenizer 决定的，而是模型架构和计算资源决定的。

```
早期 GPT-2:    1,024 Token  ≈ 一页纸
GPT-3:         4,096 Token  ≈ 三页纸
GPT-3.5:      16,384 Token  ≈ 十页纸
GPT-4 Turbo: 128,000 Token  ≈ 一本小书
Claude 3:    200,000 Token  ≈ 一本中等长度的书
Gemini 1.5: 1,000,000 Token ≈ 几本书
```

上下文窗口的含义是：模型在生成下一个 Token 时，能"看到"的之前所有 Token 的总量。如果对话历史加上当前输入超过了这个窗口，最早的部分就会被截断——模型就"忘"了开头说了什么。

想象一张这样的图：上下文窗口像一个滑动窗口，在对话历史的长条上从左到右滑动。窗口内的内容模型可以"看到"，窗口外的内容就消失了。当新的对话内容不断加入，窗口就不断向右滑动，左边最早的内容被推出窗口，被模型遗忘。

> [!NOTE]
> 扩展上下文窗口不是简单地改一个数字那么容易。原始 Transformer 的自注意力机制的计算复杂度是 O(n²)——Token 数量翻倍，计算量变成四倍。从 4K 扩展到 128K，计算量会增加上千倍。各种扩展上下文的技术（如 RoPE 外推、Flash Attention、滑动窗口注意力等）都是用来解决这个计算瓶颈的。

### Token 计费的现实意义：你的每句话都在烧钱

几乎所有主流大模型 API 都是按 Token 计费的。计费通常分两部分：

- **输入 Token（Input Tokens）**：你发给模型的内容，包括系统提示、历史对话、当前问题
- **输出 Token（Output Tokens）**：模型生成的回答

输出 Token 的价格通常是输入 Token 的 2-3 倍，因为生成比理解需要更多的计算资源。

```
一个典型的 API 调用：

系统提示（System Prompt）：200 Token
历史对话（5 轮对话）：2,000 Token
当前问题：50 Token
───────────────────────
总输入：2,250 Token
模型回答：500 Token
───────────────────────
总消耗：2,750 Token

按 GPT-4 定价计算：
输入：2,250 × $0.03/1K = $0.0675
输出：500 × $0.06/1K = $0.0300
总计：$0.0975 ≈ ¥0.7
```

一次对话不到一块钱看起来不多，但如果是一个每天服务 10 万用户的产品，每天的费用就是 7 万元，每月 210 万元。**Token 计费模式使得优化 Token 使用成为每个 AI 应用的核心工程问题。**

几个实用的 Token 节省策略：

1. **缩短系统提示**：把冗长的指令压缩成精炼的描述
2. **控制对话轮数**：及时截断过长的历史对话
3. **选择合适的模型**：简单任务用小模型（便宜），复杂任务用大模型
4. **利用中文特性**：同样意思，用英文表达通常消耗更少的 Token

### 小结

Token 是大语言模型处理文字的基本单元，介于字符和词语之间。BPE 算法通过统计和合并高频符号对来构建词表，使得任何文本都能被有效编码。不同模型的 Tokenizer 在词表大小和多语言支持上存在差异，直接影响处理效率和 API 成本。上下文窗口决定了模型一次能"看到"多少内容，而 Token 计费模式使得理解 Token 成为使用大模型的必备知识。

接下来，我们进入下一个问题：Token 变成数字 ID 之后，模型是如何让它承载"意义"的？

## §03 词嵌入——让文字变成数字的艺术
Word Embeddings: Turning Words into Numbers

### 模型不认识"猫"这个字，它只认识一个 4096 维的向量

在上一章中，我们看到"你好"被变成了 [345, 9872] 这样的整数 ID。但整数 ID 本身没有携带任何语义信息——ID 为 345 的 Token 和 ID 为 346 的 Token 之间没有任何关系，尽管它们的 ID 只差 1。

这就像用身份证号来代表人：张三的身份证号是 110101，李四的是 110102，你不能因为这两个号码挨着就说他们有什么关系。

模型需要一种方式把整数 ID 转化为**携带语义信息的数字表示**。这就是**词嵌入（Word Embedding）**要做的事情——把每个 Token 映射到一个高维向量空间中，使得语义相近的词在这个空间中的位置也相近。

### 从 One-Hot 到分布式表示：为什么需要嵌入

最直觉的编码方式是 **One-Hot 编码**：如果词表有 10 万个 Token，就用一个 10 万维的向量来表示每个 Token，这个向量中只有一个是 1，其余全是 0。

```
词表大小：5（假设只有 5 个 Token）

"猫" → [1, 0, 0, 0, 0]
"狗" → [0, 1, 0, 0, 0]
"汽车" → [0, 0, 1, 0, 0]
"高兴" → [0, 0, 0, 1, 0]
"跑步" → [0, 0, 0, 0, 1]
```

One-Hot 有两个致命问题：

**第一，完全丢失了语义关系。** 在 One-Hot 编码中，任意两个不同词的向量都是"正交"的——它们的相似度完全相同。"猫"和"狗"的相似度，与"猫"和"汽车"的相似度一样，都是零。这显然不合理。

**第二，维度灾难。** 词表有 10 万个 Token，One-Hot 向量就有 10 万维，而且几乎全是零——极度稀疏，浪费存储和计算资源。

**分布式表示（Distributed Representation）** 是解决方案。它的核心思想是：用一个小得多但每个维度都有值的稠密向量来表示每个词。

```
词表大小：5（假设），嵌入维度：3

"猫"   → [0.8, -0.2, 0.5]
"狗"   → [0.7, -0.1, 0.6]    ← 和"猫"很近
"汽车" → [-0.3, 0.9, -0.4]   ← 和"猫""狗"都很远
"高兴" → [0.1, 0.3, 0.8]
"跑步" → [0.0, 0.4, 0.7]    ← 和"高兴"比较近
```

在这个 3 维空间中，"猫"和"狗"的向量很接近（因为它们都是动物），"汽车"和它们距离很远，"高兴"和"跑步"有一定的接近度（可能因为运动和情绪有正相关性）。

> [!NOTE]
> 在实际的大语言模型中，嵌入维度远不止 3 维。GPT-2 Small 用 768 维，GPT-3 用 12,288 维，GPT-4 据推测可能使用更高维度。每个维度就像是一个"语义特征轴"，虽然我们无法直接解读每个轴的含义，但它们共同编码了丰富的语义信息。

### Word2Vec 的直觉：词向量捕捉语义关系

2013 年，Google 的 Mikolov 等人发表了 Word2Vec，这是词嵌入领域的里程碑。Word2Vec 的核心发现令人惊叹：**在训练好的词向量空间中，语义关系可以用向量运算来表达。**

最著名的例子：

```
向量("国王") - 向量("男人") + 向量("女人") ≈ 向量("王后")
```

想象一张这样的图：一个二维平面上画着几个词的向量点。"国王"在右上角，"男人"在右边，"女人"在左边。从"男人"到"国王"的箭头表示"王权"这个语义维度。如果你从"女人"出发，画一个同样方向和长度的箭头，箭头的终点恰好落在"王后"附近。

这个发现的含义是：词向量空间中存在有意义的**方向**。"性别"是一个方向，"时态"是一个方向，"国家-首都"也是一个方向。

```
向量("巴黎") - 向量("法国") + 向量("中国") ≈ 向量("北京")
向量("走路") - 向量("走") + 向量("跑") ≈ 向量("跑步")
```

Word2Vec 的训练方法也很优雅。它有两种模式：

- **CBOW（连续词袋）**：用上下文词预测中间词。就像完形填空："今天天气[___]"，根据"今天""天气"来预测空格处的词。
- **Skip-gram**：用中间词预测上下文词。反过来，给你一个词，让你猜它周围可能出现哪些词。

两种方式本质上都是在利用**"分布相似性"**假设：出现在相似上下文中的词，具有相似的语义。这个假设来自语言学中的经典名言："You shall know a word by the company it keeps"（你可以通过一个词的伙伴来认识它）。

> [!TIP]
> Word2Vec 的直觉不仅适用于词语。后来的研究发现，类似的嵌入思想可以应用于任何离散对象——用户行为、商品、图片、音乐。只要对象之间存在某种"共现"或"相似"关系，就可以用嵌入方法将其映射到连续向量空间中。这是整个推荐系统和表示学习的核心思想。

### 上下文嵌入 vs 静态嵌入：同一个词在不同语境中为什么不同

Word2Vec 有一个根本性的局限：**每个词只有一个固定的向量**。"苹果"无论是作为水果还是作为公司，在 Word2Vec 中都是同一个向量。但显然，"我吃了一个苹果"和"苹果发布了新手机"中的"苹果"含义完全不同。

这就是**静态嵌入（Static Embedding）**和**上下文嵌入（Contextual Embedding）**的区别：

```
静态嵌入（Word2Vec）：
"苹果" → [0.5, 0.3, -0.2, ...]   （永远不变）

上下文嵌入（Transformer）：
"我吃了一个苹果" 中的 "苹果" → [0.6, 0.2, -0.1, ...]  （倾向水果）
"苹果发布了新手机" 中的 "苹果" → [-0.3, 0.8, 0.5, ...] （倾向公司）
```

上下文嵌入的实现方式是：Token 的初始嵌入只是"起点"，随后它会经过 Transformer 的多层处理，每一层的自注意力机制都会根据周围的词来"调整"这个 Token 的表示。

想象一张这样的图：一个 Token 的嵌入向量从底部输入，经过第 1 层注意力后，吸收了周围词的信息；经过第 2 层后，吸收了更大范围的信息；经过第 3 层后，它已经"看到"了整句话……最终输出的向量不仅包含了"苹果"本身的语义，还融合了整个句子上下文的信息。

这就是为什么大语言模型能理解"苹果"在不同语境中的不同含义——不是因为它查了什么词典，而是因为**上下文信息被直接编码进了向量表示中**。

```
处理流程：

Token ID:  [1567]                    （"苹果"的 ID）
    ↓ 查嵌入矩阵
初始嵌入:  [0.5, 0.3, -0.2, ...]    （静态，只依赖 ID）
    ↓ Transformer Layer 1
第1层表示: [0.55, 0.28, -0.15, ...] （开始融合上下文）
    ↓ Transformer Layer 2
第2层表示: [0.6, 0.2, -0.1, ...]    （更多上下文信息）
    ↓ ... 更多层 ...
最终表示:  [0.6, 0.2, -0.1, ...]    （完整的上下文嵌入）
```

### 嵌入维度怎么选：128 vs 1024 vs 4096 的区别

嵌入维度是模型设计中的一个关键超参数。维度越高，能编码的语义信息越丰富，但计算成本也越大。

| 嵌入维度 | 代表模型 | 参数量级 | 典型能力 |
|---------|---------|---------|---------|
| 128 | 早期 Word2Vec | 百万 | 基本语义关系 |
| 768 | GPT-2 Small, BERT-Base | 亿 | 复杂语义理解 |
| 4,096 | LLaMA-2 70B | 百亿-千亿 | 深度推理能力 |
| 12,288 | GPT-3 175B | 千亿 | 高级推理、创作 |

维度太低，模型无法捕捉复杂的语义关系，就像用只有 3 个数字来描述一个人的外貌——信息量不够。维度太高，模型可能过拟合训练数据，而且计算和存储成本指数增长。

实际中的选择是经验和实验的结果。目前主流大模型使用的嵌入维度在 4096 到 12288 之间。

### 位置编码：Transformer 没有 RNN 的顺序感，靠什么知道词序

前面讨论的嵌入只编码了"这个词是什么"（语义），但没有编码"这个词在句子中的哪个位置"（位置）。对 RNN 来说，位置信息天然存在——它是按顺序处理词的，第 3 个词自然是在第 3 步被处理的。但 Transformer 的自注意力机制是完全并行的——它同时处理所有词，**天生不知道词的顺序**。

这就好比一个学生拿到了一篇论文，但论文的所有词都被打乱了排列在桌上。他认识每个词，但不知道它们原来的顺序，也就无法理解论文的内容。

**位置编码（Positional Encoding）** 就是给每个 Token 的嵌入向量加上一个"位置标签"，告诉模型"这个词出现在第几个位置"。

想象一张这样的图：一行 Token 排列着，每个 Token 下面有一个标记它位置的标签。位置标签和语义嵌入加在一起，形成最终的输入向量。

```
位置：     1          2          3          4
Token：   "我"       "爱"       "北京"     "天安门"
语义嵌入： [0.5,...]  [0.3,...]  [0.8,...]  [0.6,...]
位置编码： [0.1,...]  [0.2,...]  [0.3,...]  [0.4,...]
─────────────────────────────────────────────────
最终输入： [0.6,...]  [0.5,...]  [1.1,...]  [1.0,...]
```

**正弦位置编码（Sinusoidal Positional Encoding）**

原始 Transformer 论文中使用的方法。它用不同频率的正弦和余弦函数来编码位置：

- 低频分量编码"大致位置"（靠前、靠后）
- 高频分量编码"精确位置"（第几个）

想象音乐中的和弦：低音给出整体的节奏框架，高音给出精确的旋律细节。正弦位置编码类似——不同频率叠加在一起，为每个位置生成一个独特的"指纹"。

正弦编码的优点是：理论上可以外推到训练时没见过的更长的序列。但实际上这种外推能力很有限。

**旋转位置编码（RoPE, Rotary Position Embedding）**

RoPE 是目前主流大模型（LLaMA、Mistral、Qwen 等）采用的位置编码方案。它的核心思想非常巧妙：**不是把位置信息加到向量上，而是通过旋转向量来编码位置。**

想象一张这样的图：一个二维平面上的向量，随着位置的增加，向量被一点点旋转。位置 1 的向量旋转了一个小角度，位置 2 的向量旋转了两倍的角度，以此类推。两个位置的相对距离，可以通过它们旋转角度的差来恢复。

RoPE 的关键优势：

1. **相对位置感知**：模型可以直接感知两个 Token 之间的相对距离，而不仅仅是绝对位置
2. **更好的外推性**：可以更容易地扩展到比训练时更长的序列
3. **计算效率高**：可以直接融入注意力计算，不需要额外的参数

> [!NOTE]
> 位置编码的选择直接影响了模型的上下文窗口能力。RoPE 配合一些扩展技术（如 YaRN、NTK-aware scaling），可以让模型在训练时只用了 4K Token 的上下文，但在推理时能处理 128K 甚至更长的序列。这也是为什么 2023-2024 年间，各家模型的上下文窗口从 4K 迅速扩展到 128K 甚至百万级。

### 小结

词嵌入是将离散的文字转化为连续向量表示的技术，使得语义相近的词在向量空间中的位置也相近。从 One-Hot 的稀疏表示到分布式表示的稠密向量，再到 Transformer 的上下文嵌入，每一步都在让模型对文字的"理解"更精细。位置编码则解决了 Transformer 并行处理时缺乏顺序信息的问题，从正弦编码到 RoPE 的演进让模型能够处理越来越长的上下文。

现在，我们有了 Token 和嵌入这两个基础构件。接下来就是重头戏：Transformer——把这些构件组装成强大引擎的架构。

## §04 Transformer 架构——注意力就是一切
The Transformer: Attention Is All You Need

### 2017 年的一篇论文改变了整个 AI 行业

2017 年 6 月，Google 的八位研究者发表了一篇名为 "Attention Is All You Need" 的论文。这篇论文提出了一种新的神经网络架构——Transformer。当时的背景是，机器翻译领域的主流方法是基于 RNN 的 Seq2Seq 模型，而 Transformer 用纯注意力机制取代了循环结构，在翻译质量上大幅超越了当时的最优系统。

没有人预料到，这篇最初针对机器翻译的论文，会在随后几年成为整个 AI 行业的基础架构。从 BERT 到 GPT-4，从 DALL-E 到 Sora，几乎所有当前最先进的 AI 模型都建立在 Transformer 架构之上。

### Transformer 整体架构：从 Encoder-Decoder 到 Decoder-Only

原始 Transformer 是一个 **Encoder-Decoder** 架构，专门为机器翻译设计：

想象一张这样的图：左侧是 Encoder（编码器），右侧是 Decoder（解码器），中间通过注意力连接。输入的英文句子从左侧进入 Encoder，经过多层处理后生成一组"编码表示"；Decoder 基于这组编码表示，逐词生成翻译后的法文句子。

```
原始 Transformer（用于翻译）：

Encoder:                    Decoder:
"I love AI"                "<BOS> j' adore"  →  "l'"  →  "intelligence"  → ...
    ↓                           ↓
[Embedding]                 [Embedding]
    ↓                           ↓
[Multi-Head Attention]      [Masked Attention]  ← 只能看到已生成的词
    ↓                           ↓
[FFN]                       [Cross-Attention]   ← 看 Encoder 的输出
    ↓                           ↓
[× N 层]                    [FFN]
                                ↓
                            [× N 层]
                                ↓
                            [输出概率分布]
```

但随着研究深入，人们发现对于语言建模（即预测下一个词），**只需要 Decoder 部分就够了**。这就是 **Decoder-Only 架构**——GPT 系列和大多数现代大语言模型的选择。

为什么 Decoder-Only 够用？因为语言模型的核心任务是"根据已看到的文本，预测下一个词"，这天然就是一个从左到右的生成过程，不需要 Encoder 来编码什么。输入和输出都是同一种语言的文本，Decoder 自己就能处理。

```
Decoder-Only 架构（GPT 等）：

输入："从前有座山，"
         ↓
    [Token Embedding + Positional Encoding]
         ↓
    [Transformer Block × N]
         │
         ├→ [Masked Multi-Head Attention]
         ├→ [Add & Norm]
         ├→ [FFN]
         └→ [Add & Norm]
         ↓
    [线性层 + Softmax]
         ↓
    输出概率：P("山") = 0.35, P("里") = 0.25, ...
```

> [!NOTE]
> 除了 Decoder-Only，还有 Encoder-Only 架构（如 BERT），它在理解类任务（分类、标注）上表现出色，但不擅长生成文本。目前大语言模型的主流选择是 Decoder-Only，因为它在生成任务上的优势最明显，而且通过适当的训练方法也能胜任理解类任务。

### 一句话看懂 Transformer：嵌入 → 多层注意力 + FFN → 输出概率

如果把 Transformer 的复杂架构压缩成一句话：

**输入的 Token 经过嵌入层变成向量，然后经过 N 个 Transformer Block（每个 Block 包含一个注意力层和一个前馈网络），最后通过线性层输出下一个 Token 的概率分布。**

就这么简单。Transformer 的复杂性不在于架构的
![图4-1：Transformer Decoder-Only架构全景图](images/transformer-architecture.png)
"形状"，而在于每个组件内部的设计精巧。

### 每一层的结构：残差连接、Layer Norm、前馈网络

每个 Transformer Block 由以下组件按顺序组成：

```
输入
 │
 ├→ Multi-Head Self-Attention
 │        ↓
 ├→ Add（残差连接：把输入直接加到注意力输出上）
 │        ↓
 ├→ Layer Norm（层归一化：稳定数值范围）
 │        ↓
 ├→ FFN（前馈网络：两层全连接 + 激活函数）
 │        ↓
 ├→ Add（残差连接：把 Layer Norm 的输出加到 FFN 输出上）
 │        ↓
 └→ Layer Norm
          ↓
        输出
```

让我们逐一理解这些组件。

**残差连接（Residual Connection）**

残差连接的做法极其简单：把模块的输入直接加到模块的输出上。

```
输出 = 模块(输入) + 输入
```

为什么要这么做？当模型有很多层（比如 96 层）时，反向传播的梯度信号在逐层传递时会逐渐衰减，导致靠近输入的层几乎无法被训练。残差连接提供了一条"高速公路"，让梯度可以直接跳过中间层，从输出端直接传到输入端。

想象一张这样的图：一条主干道从底部直通顶部，沿途有多个收费站（各层处理）。残差连接就像一条平行的快速通道，信息可以选择走快速通道跳过某些收费站。即使某些层暂时没学到有用的东西，信息也不会丢失，因为快速通道保证了信号的传递。

**Layer Norm（层归一化）**

Layer Norm 的作用是**稳定每一层的数值范围**。它对每个样本的特征向量做归一化，使其均值为 0、方差为 1。

为什么需要它？在深层网络中，每一层的输出被下一层作为输入，微小的数值偏移会随着层数被不断放大（内部协变量偏移问题），导致训练不稳定。Layer Norm 就像每个关卡的质量检查员，确保数值始终在合理范围内。

> [!TIP]
> Layer Norm 有两个主流变体：Post-LN（先 Attention 再 Norm，原始 Transformer 使用）和 Pre-LN（先 Norm 再 Attention，现代大模型常用）。Pre-LN 的训练稳定性更好，是目前的主流选择。

**前馈网络（FFN, Feed-Forward Network）**

每个 Transformer Block 中的 FFN 是一个两层的全连接网络：

```
FFN(x) = W₂ · activation(W₁ · x + b₁) + b₂
```

其中 activation 通常是 ReLU（Rectified Linear Unit，修正线性单元）或 GELU（Gaussian Error Linear Unit，高斯误差线性单元）激活函数。FFN 的一个重要特点是：中间层的维度通常是输入维度的 4 倍。例如，如果嵌入维度是 4096，FFN 的中间层就是 16384 维——先"展开"，再"压缩"回去。

FFN 的作用可以理解为**知识的存储和检索**。如果说注意力层负责让 Token 之间互相交流信息，那么 FFN 就负责根据这些信息做进一步的加工和变换。研究表明，FFN 的权重矩阵中编码了大量的**事实性知识**——"法国的首都是巴黎"这类知识主要存储在 FFN 中。

想象一张这样的图：注意力层像是一个会议室，所有 Token 围坐在一起讨论；FFN 像是每个 Token 会后的独立思考时间，根据讨论内容查阅自己的笔记（权重矩阵），形成更深入的理解。

### 为什么 GPT 选择 Decoder-Only 架构

当前主流大语言模型（GPT、LLaMA、Mistral、Qwen）几乎全部选择了 Decoder-Only 架构。这并非偶然，而是基于几个关键优势：

**第一，训练效率更高。** Decoder-Only 架构中，每个训练样本的每个 Token 都可以作为一个训练目标——给定前 N 个 Token 预测第 N+1 个。而 Encoder-Decoder 架构通常只有解码端的 Token 作为训练目标。这意味着同样的数据量下，Decoder-Only 能提取更多的训练信号。

**第二，架构更简单。** 没有 Encoder-Decoder 之间的交叉注意力，整个模型只有一种类型的 Block，更容易扩展和优化。

**第三，统一了理解和生成。** 虽然 Decoder-Only 天然擅长生成，但通过合适的训练方式（如在输入中提供上下文），它也能很好地完成理解任务。这种统一性简化了整个技术栈。

**第四，KV Cache 的天然适配。** 这是下一节要讨论的重点——Decoder-Only 的因果注意力使得推理时的 KV Cache 实现非常高效。

### 从单层到多层：深度带来的能力跃迁

Transformer 的层数（深度）是决定模型能力的关键维度之一。每增加一层，模型就能进行更深层次的抽象和推理。

| 模型 | 层数 | 嵌入维度 | 参数量 |
|------|------|---------|--------|
| GPT-2 Small | 12 | 768 | 1.24 亿 |
| GPT-2 XL | 48 | 1,600 | 15 亿 |
| GPT-3 | 96 | 12,288 | 1,750 亿 |
| LLaMA-2 70B | 80 | 8,192 | 700 亿 |

层数越多，模型能捕捉的模式就越复杂。浅层可能学到基本的语法关系，中层学到语义关系，深层学到逻辑推理和长距离依赖。这是一个逐层抽象的过程——就像人类理解一篇文章时，先理解字词，再理解句子，然后理解段落，最后理解全文的中心思想。

> [!WARNING]
> 层数不是越多越好。过深的网络会面临梯度退化、训练不稳定、推理延迟增加等问题。通常需要在模型能力和计算效率之间取得平衡。一些研究也在探索"浅而宽"的架构是否可以达到与"深而窄"架构相当的效果。

### Multi-Head Attention：为什么需要多个头

自注意力机制使用一组 Q、K、V 来计算注意力。但一个问题是：一句话中存在多种不同类型的关系。

```
"猫坐在垫子上，它看起来很舒服"
```

在这句话中，"它"和"猫"之间存在指代关系，"坐"和"垫子"之间存在空间关系，"舒服"和"猫"之间存在状态关系。单一的注意力头很难同时捕捉所有这些不同类型的关系。

**Multi-Head Attention 的解决方案**：使用多组不同的 Q、K、V（每个"头"有自己的参数），让每个头关注不同的关系类型，然后把所有头的输出拼接起来。

```
单头注意力：
"它" 用一组 Q/K/V → 捕捉一种关系模式

多头注意力（8 头）：
Head 1："它" 用第 1 组 Q/K/V → 可能关注指代关系
Head 2："它" 用第 2 组 Q/K/V → 可能关注空间关系
Head 3："它" 用第 3 组 Q/K/V → 可能关注情感关系
...
Head 8："它" 用第 8 组 Q/K/V → 可能关注语法关系

最终：拼接 8 个头的输出 → 综合多种关系模式
```

想象一张这样的图：一个 Token 的嵌入向量被投影到 8 个不同的子空间中，每个子空间中独立计算注意力，得到 8 个不同的"视角"。这 8 个视角像 8 个手电筒，从不同角度照亮同一个场景，最后把 8 个视角的信息合并起来，形成对这个 Token 在上下文中角色的全面理解。

> [!TIP]
> 多头注意力的每个头的维度是 总维度/头数。例如，嵌入维度 4096，8 个头，则每个头的维度是 512。总计算量与单头注意力基本相同，但信息更丰富。这也是 Multi-Head Attention 的巧妙之处——不增加计算成本，却显著增强了表达能力。

### KV Cache：推理加速的关键技巧

Transformer 在推理（生成文本）时是逐 Token 进行的：每次生成一个 Token，把它加到已有序列末尾，再预测下一个。问题在于，每生成一个新 Token，注意力机制都要重新计算**所有**之前 Token 的 Q、K、V 值。对于 128K 的上下文，这意味着大量重复计算。

**KV Cache 的核心思想**：既然已经计算过的 Token 的 Key 和 Value 不会变（因为它们不受后续 Token 的影响），那就把它们缓存起来，下次只需要计算新 Token 的 Q、K、V，然后用新 Token 的 Q 去查询缓存中所有旧的 K、V。

```
没有 KV Cache（每步重新计算全部）：
生成第 1 个 Token：计算 1 个 Token 的 Q/K/V
生成第 2 个 Token：重新计算 2 个 Token 的 Q/K/V
生成第 3 个 Token：重新计算 3 个 Token 的 Q/K/V
...
生成第 N 个 Token：重新计算 N 个 Token 的 Q/K/V
总计算量：1 + 2 + 3 + ... + N = O(N²)

有 KV Cache（缓存已计算的 K/V）：
生成第 1 个 Token：计算 1 个 Token 的 Q/K/V，缓存 K/V
生成第 2 个 Token：只计算 1 个新 Token 的 Q/K/V，复用缓存的 K/V
生成第 3 个 Token：只计算 1 个新 Token 的 Q/K/V，复用缓存的 K/V
...
总计算量：N × O(1) = O(N)
```

KV Cache 将生成 N 个 Token 的注意力计算复杂度从 O(N²) 降低到 O(N)，这是一个巨大的加速。在实际系统中，KV Cache 能让推理速度提升 10-100 倍。

想象一张这样的图：一个不断增长的柜子（缓存），每生成一个新 Token，就把它的 Key 和 Value 存入柜子。下一个 Token 来了，只需要算自己的 Query，然后去柜子里查所有已有的 Key-Value 对。

KV Cache 也带来了一个实际问题：**显存占用**。对于 LLaMA-2 70B 模型，在 4096 Token 的上下文长度下，KV Cache 需要占用约 16GB 显存——几乎和模型权重本身一样多。这也是为什么各家都在研究更高效的缓存策略，如 GQA（Grouped Query Attention）和 MQA（Multi-Query Attention），通过共享 Key 和 Value 头来减少缓存大小。

| 优化方法 | Key/Value 头数 | KV Cache 大小 | 使用模型 |
|---------|---------------|--------------|---------|
| MHA（标准多头） | 64 头 | 100% | GPT-3 |
| GQA（分组查询） | 8 头 | 12.5% | LLaMA-2 70B |
| MQA（多查询） | 1 头 | 1.5% | Mistral |

### 小结

Transformer 架构是当前大语言模型的基础。它的核心是一个简洁的流水线：嵌入 → 多层注意力 + FFN → 输出概率。每个 Transformer Block 包含自注意力层（让 Token 之间交换信息）、前馈网络（加工和存储知识）、残差连接（保证梯度流动）和层归一化（稳定训练）。现代大语言模型几乎全部采用 Decoder-Only 架构，因为它在训练效率、推理加速（KV Cache）和架构简洁性上都有优势。Multi-Head Attention 让模型能同时关注多种关系，而 KV Cache 则是推理加速的关键工程技巧。

到这一章为止，我们已经拆解了大语言模型的所有基础构件——从 Token 到嵌入，从注意力到 Transformer。在下一部分中，我们将深入模型的训练过程，看看这些构件是如何被海量数据"唤醒"的。

## §05 自注意力机制——捕捉全局关联的秘密
Self-Attention: Capturing Global Relationships

### "它"到底指什么？

看这句话："猫咪坐在垫子上，**它**看起来很舒服。"

作为人类，你毫不犹豫地知道"它"指的是"猫咪"而不是"垫子"。但你是怎么知道的？你并不是靠什么语法规则一步步推导——你是在一瞬间，把"它"和前面所有词进行了关联比较，然后发现"猫咪"和"舒服"的语义搭配最自然。

这就是**自注意力机制**要做的事情：让模型学会在处理每一个词时，自动去"看"句子里的其他所有词，判断谁跟自己的关系最密切。

在 Transformer 出现之前，循环神经网络（RNN）处理句子的方式像读书——从左到右一个词一个词地读，读到句末时，开头的词可能已经"忘"了。自注意力机制则完全不同：**它让每个词都能直接看到所有其他词，无论距离多远。** 这是 Transformer 能取代 RNN 的根本原因。

### Query、Key、Value：图书馆检索的直觉

自注意力机制的核心概念是三个向量：**Query（查询）、Key（键）、Value（值）**。这三个名字来自信息检索领域，我们可以用一个图书馆的类比来理解。

想象你去图书馆找书：

- **Query**：你心里想找什么——"我想找一本关于 Transformer 的技术书"。这是你的**查询意图**。
- **Key**：每本书的书脊上写着什么——书名、作者、分类标签。这是每本书的**身份信息**。
- **Value**：书的具体内容——你选中一本书后实际读到的东西。这是书的**实际信息**。

找书的过程是这样的：

1. 你拿着自己的 Query，和书架上每本书的 Key 做比较
2. 匹配度高的书获得高"注意力分数"
3. 你最终得到的是所有书的 Value 按匹配度加权后的综合结果

在自注意力中，每个词都会生成自己的 Q、K、V 三个向量。
![图5-1：自注意力机制Q/K/V计算流程](images/self-attention-qkv.png)
然后每个词用自己的 Q 去和所有词的 K 做匹配，得到注意力分数，再用这些分数对 V 做加权求和。

```
词 → 线性变换 → Query (我在找什么)
               → Key   (我是什么)
               → Value (我的内容)
```

> [!NOTE]
> Q、K、V 是通过对输入向量做线性变换得到的。具体来说，就是分别乘以三个不同的权重矩阵 W_Q、W_K、W_V。这些矩阵的值在训练过程中学出来。

### 注意力计算的四个步骤

让我们用一个具体例子走一遍完整流程。假设输入句子是："猫 坐 在 垫子 上"。

**第一步：线性变换——生成 Q、K、V**

每个词的嵌入向量分别乘以三个权重矩阵，得到该词的 Query、Key、Value 向量：

```
"猫"   → Q₁, K₁, V₁
"坐"   → Q₂, K₂, V₂
"在"   → Q₃, K₃, V₃
"垫子" → Q₄, K₄, V₄
"上"   → Q₅, K₅, V₅
```

**第二步：点积——计算相关性**

用每个词的 Query 和所有词的 Key 做点积，得到注意力分数。以"猫"为例：

```
Score(猫, 猫)   = Q₁ · K₁  → 某个数值，比如 3.2
Score(猫, 坐)   = Q₁ · K₂  → 比如 2.8
Score(猫, 在)   = Q₁ · K₃  → 比如 0.5
Score(猫, 垫子) = Q₁ · K₄  → 比如 0.3
Score(猫, 上)   = Q₁ · K₅  → 比如 0.1
```

点积值越大，说明两个词的关联性越强。

**第三步：Softmax——归一化为概率**

把分数除以一个缩放因子（通常是 Key 维度的平方根），然后通过 Softmax 函数，把一组数字转化为概率分布（所有值在 0 到 1 之间，总和为 1）：

```
缩放后的分数: [3.2, 2.8, 0.5, 0.3, 0.1] / √d_k
Softmax 后:    [0.45, 0.35, 0.10, 0.07, 0.03]
```

这个结果就是**注意力权重**——"猫"在处理自己时，对每个词投了多少"注意力"。

**第四步：加权求和——得到输出**

用注意力权重对 Value 向量做加权求和，得到"猫"这个词经过自注意力处理后的新表示：

```
Output(猫) = 0.45 × V₁ + 0.35 × V₂ + 0.10 × V₃ + 0.07 × V₄ + 0.03 × V₅
```

> [!TIP]
> 把这四个步骤串起来，就是自注意力的完整计算。整个过程没有任何递归或循环——所有位置的计算都是并行的，这正是 Transformer 比 RNN 快的根本原因。

### 为什么点积能衡量相关性

点积衡量相关性的背后是一个几何直觉。

想象每个词是一个高维空间中的箭头（向量）。两个箭头的点积等于它们的长度乘以夹角的余弦值。如果两个向量方向相近（夹角小），点积就大；方向相反（夹角接近 180 度），点积为负；完全垂直，点积为零。

在注意力机制中，Query 和 Key 经过了各自的线性变换——也就是说，它们被投影到了不同的"视角空间"中。训练过程会学习到这样的权重：语义相关的词在投影后的空间中方向会更一致，点积值会更高。

这就像你在图书馆找书：你的查询意图（Query）和书的关键词（Key）如果能对上——即它们在同一个"语义空间"中方向接近——匹配度就高。

> [!NOTE]
> 缩放因子（除以 √d_k）的存在是因为：当向量维度很高时，点积的值会变得很大，导致 Softmax 输出接近 one-hot（一个值接近 1，其他接近 0），梯度几乎为零，训练无法有效进行。除以 √d_k 可以保持梯度稳定。这就是为什么这个机制叫做 **Scaled Dot-Product Attention**。

### Multi-Head Attention：多个视角看同一句话

单头注意力只有一个"视角"。但语言的关系是复杂的——一句话中可能同时存在多种关联：语法关系、语义关系、指代关系、位置关系……

**多头注意力**的做法是：不只用一组 Q/K/V 权重，而是同时用多组（比如 8 组、16 组甚至 32 组），每组独立计算注意力，然后把结果拼接在一起，再做一次线性变换得到最终结果。

用读书类比：这就像让 8 个不同背景的人同时读一句话——一个关注语法结构，一个关注情感色彩，一个关注指代关系，一个关注实体信息……最后把他们各自的发现汇总。

```
                    ┌── Head 1: Q₁K₁V₁ → 注意力1 ──┐
输入 ──→ 分成8组 ──┼── Head 2: Q₂K₂V₂ → 注意力2 ──┼── 拼接 ──→ 线性变换 ──→ 输出
                    ├── ...                          │
                    └── Head 8: Q₈K₈V₈ → 注意力8 ──┘
```

在实践中，如果总维度是 512，8 个头，则每个头处理 512/8 = 64 维。头数越多，模型能捕捉的关系类型就越丰富，但每个头的维度会变小，单头表达能力会下降。这是一个需要权衡的设计选择。

> [!WARNING]
> 多头注意力并不意味着每个头都会学到人类可解释的特定功能（比如"这个头专门处理指代"）。实际情况更加复杂——头与头之间的分工是在训练过程中自然涌现的，不同头可能关注不同的模式，但这些模式不一定能被简单地命名。

### 因果注意力：为什么生成模型只能看"过去"

GPT 这类生成模型的工作方式是：根据已有的词，预测下一个词。这意味着在预测第 t 个词时，模型只能看到第 1 到 t-1 个词——它不能"偷看"未来。

这在注意力机制中通过**因果掩码（Causal Mask）**实现。

具体做法是：在计算注意力分数后，用一个上三角为负无穷的矩阵"遮住"未来位置的分数：

```
注意力分数矩阵（5个词的例子）：

        猫    坐    在    垫子  上
猫     [ 3.2, -∞,   -∞,   -∞,  -∞ ]
坐     [ 2.8,  2.1, -∞,   -∞,  -∞ ]
在     [ 0.5,  1.0,  1.8, -∞,   -∞ ]
垫子   [ 0.3,  0.7,  1.5,  2.5, -∞ ]
上     [ 0.1,  0.4,  0.9,  2.0,  1.2 ]
```

负无穷经过 Softmax 后变成 0，这样每个位置就完全看不到后面的词了。

为什么要这么设计？想想人类说话——你说每一句话时，也只能基于之前说过的内容和心中的想法，不能用到还没说出来的话。因果注意力模拟的正是这个过程。

> [!TIP]
> BERT 使用的是**双向注意力**（不遮掩），所以它能同时看到一句话的前后文。这使它更擅长"理解"任务（如分类、命名实体识别），但不适合文本生成。GPT 使用因果注意力，更适合生成任务。这是一个重要的架构分叉。

### 注意力模式可视化：模型到底在关注什么

训练好的模型，其注意力权重是可以直接观察的。研究人员通过可视化注意力矩阵，发现了很多有趣的模式。

以句子 "The animal didn't cross the street because **it** was too tired." 为例，在 Transformer 的某一层中，注意力可视化显示"it"这个词把大部分注意力投给了"animal"——模型确实学会了正确的指代消解。

但要注意：

1. **不同层关注不同粒度的关系。** 底层可能关注相邻词的语法关系，高层关注句子级别的语义关系。
2. **不同头关注不同的模式。** 有些头关注相邻词，有些头关注标点符号，有些头关注特定距离的词。
3. **注意力权重不等于重要性。** 一个词对另一个词的注意力高，不代表它"更重要"——这只是模型内部信息路由的一种方式。

> [!WARNING]
> 注意力可视化是理解模型行为的工具，但不要过度解读。注意力权重只是模型计算的中间结果，高权重并不必然意味着该位置对最终输出起了决定性作用。最近的研究表明，注意力权重和模型输出的因果关系比我们直觉上认为的要弱。

### 本章小结

自注意力机制是 Transformer 的核心创新，它解决了序列建模中一个根本问题：**如何高效地建模任意距离的词间关系**。

关键要点：

- **Q/K/V 机制**通过"查询-匹配-取值"的信息检索范式，让每个词都能从所有其他词中提取相关信息
- **点积注意力**用几何直觉衡量相关性，加上缩放因子保证训练稳定
- **多头注意力**让模型同时从多个"视角"分析语言关系
- **因果掩码**确保生成模型不会"看到未来"
- 注意力计算是**完全并行的**，这是 Transformer 相比 RNN 的巨大工程优势

自注意力机制回答了"模型如何理解词与词之间的关系"这个问题。但光有机制不够——模型还需要大量数据来"练习"。下一章我们就来看预训练：模型如何从海量文本中学习世界知识。

## §06 预训练——从海量文本中学习世界知识
Pre-training: Learning World Knowledge from Text

### 20 万年的阅读量

GPT-4 的训练数据总量没有官方确切数字，但据各种估计，大约在 13 万亿（13 Trillion）Token 的量级。做个简单换算：一个英语母语的大学毕业生一辈子大概读过 1 亿词，而 GPT-4 的训练数据相当于**一个人读 20 万年**的内容。

这些数据涵盖了互联网网页、维基百科、书籍、学术论文、GitHub 代码仓库……几乎所有人类数字化知识的精华都在其中。预训练就是让模型"读"完所有这些内容，从中学习语言的规律、世界的知识、推理的模式。

这一章我们来拆解预训练的完整流程：数据从哪来、怎么清洗、怎么训练、代价有多大。

### 预训练的目标：下一个词预测

预训练的核心任务极其简单：**给定前面的词，预测下一个词。**

给它 "今天天气真"，它要预测 "好" 的概率高于 "差"。
给它 "法国的首都是巴"，它要预测 "黎" 的概率最高。
给它 "2 + 2 ="，它要预测 "4" 的概率最高。

这个看似简单的任务，实际上要求模型学会：

- **语法结构**："一只猫___在垫子上" → "躺" 比 "躺躺躺" 更可能
- **事实知识**："地球绕着___转" → "太阳" 而不是 "月亮"
- **推理能力**："如果下雨，地面会___" → "湿" 需要因果推理
- **上下文理解**："苹果公司发布了新___" → "产品" 而不是 "水果"

当训练数据量达到万亿 Token 的规模，这种"预测下一个词"的训练就会迫使模型发展出相当复杂的内部表示——因为只有真正"理解"了语言的深层结构，才能在各种语境下都做出准确的预测。

> [!NOTE]
> 这被称为"下一个词预测（Next Token Prediction）"或"语言建模（Language Modeling）"。模型的目标不是记住所有文本，而是学习文本背后的统计规律——这些规律本质上编码了人类的语言知识和世界知识。

### 训练数据从哪来

大模型的训练数据主要来自以下几个来源：

**Common Crawl —— 互联网的快照**

Common Crawl 是一个非营利组织，定期爬取整个互联网的网页并免费开放。它的数据量是 PB 级别的——如果你把它全部下载下来，大概需要几十个硬盘。这是大多数大模型训练数据的主要来源，通常占总量的 50%-80%。

**Wikipedia —— 高质量百科全书**

维基百科的数据量相对较小（大约几十亿 Token），但质量极高。它是经过人工审核的结构化知识，涵盖几乎所有学科。几乎所有大模型都会把 Wikipedia 作为核心训练数据。

**书籍 —— 系统性知识**

书籍提供系统性、深度性的知识——一本教材的知识结构远比维基百科条目完整。训练数据中通常会包含大量电子书，涵盖教科书、小说、非虚构类作品等。

**代码 —— GitHub 等代码仓库**

代码不仅是"另一种语言"，它还天然包含逻辑推理和结构化思维。研究发现，训练数据中包含代码会显著提升模型的推理能力。GPT-4 的训练数据中就包含了大量 GitHub 代码。

**学术论文 —— arXiv 等**

学术论文提供了严谨的学术写作范本和前沿知识。特别是数学和理工科论文，对提升模型的推理能力很有帮助。

**其他来源**：Reddit 讨论帖（提供对话风格）、Stack Overflow（技术问答）、新闻文章等。

| 数据来源 | 特点 | 大致占比 |
|----------|------|----------|
| Common Crawl | 量大质杂 | 50%-80% |
| Wikipedia | 量小质高 | 3%-5% |
| 书籍 | 系统深度 | 5%-10% |
| 代码 | 逻辑性强 | 5%-15% |
| 学术论文 | 严谨专业 | 2%-5% |
| 其他 | 多样性补充 | 剩余部分 |

> [!WARNING]
> 以上比例是大致估计，各家公司对训练数据的具体构成严格保密。OpenAI 从 GPT-3 开始就不再公开训练数据细节，Google 和 Anthropic 也同样保密。业界将这些细节视为核心商业机密。

### 数据清洗流水线

互联网上的原始数据充斥着噪音——垃圾信息、重复内容、低质量文本、有害信息。如果直接把 Common Crawl 的原始数据喂给模型，效果会非常差。数据清洗是预训练中最关键（也最容易被忽视）的环节之一。

一个典型的数据清洗流水线包括以下步骤：

**第一步：提取文本**

从 HTML 网页中提取纯文本内容，去除导航栏、广告、页脚等无用内容。这看似简单，实际上是数据处理中的一大挑战——网页的结构千差万别，提取工具需要处理各种边缘情况。

**第二步：去重**

互联网上存在大量重复内容——同一篇文章被多个网站转载、模板化的页面内容等。研究表明，去重对模型质量的影响巨大。常用的去重方法包括：

- **精确去重**：完全相同的文档直接删除副本
- **模糊去重**：用 MinHash 等算法检测相似但不完全相同的文档
- **文档级去重**：在文档粒度上去重
- **段落级去重**：更细粒度，避免不同文档中出现相同段落

Meta 训练 LLaMA 模型时，使用了一个重要的发现：**训练数据的去重程度对模型性能的影响，几乎等同于把模型参数量翻倍。** 这说明数据质量比数据量更重要。

**第三步：质量过滤**

去除低质量文本，方法包括：

- **语言过滤**：只保留目标语言的文本
- **长度过滤**：去除过短（如只有几个词）或异常长的文档
- **启发式过滤**：基于规则去除乱码、模板文本、日志文件等
- **分类器过滤**：训练一个分类器来识别高质量文本（类似于用 Wikipedia 和书籍作为正样本，用随机网页作为负样本）
- **困惑度过滤**：用一个已训练好的小语言模型计算文本的困惑度，过高（模型完全看不懂）或过低（过于模板化）的文本被过滤掉

**第四步：有害内容过滤**

去除包含色情、暴力、个人隐私信息（如身份证号、电话号码）等有害内容的文本。这不仅是为了模型安全，也是为了合规。

**第五步：数据混合与采样**

不同来源的数据质量差异很大，不能简单合并。通常的做法是：高质量数据（如 Wikipedia、书籍）的采样概率会被提高，低质量数据（如随机网页）会被降采样。例如，虽然 Common Crawl 占原始数据的 80%，但在最终训练数据中可能只占 60%，而 Wikipedia 虽然只占原始数据的 3%，在训练中可能被加权到 8%。

```
原始数据 → 文本提取 → 去重 → 质量过滤 → 有害内容过滤 → 数据混合 → 训练数据
  PB级                                       TB级                        万亿Token级
```

> [!TIP]
> 数据工程是大模型训练中最被低估的环节。很多人以为大模型的成功主要靠算法和算力，但实际上，精心设计的数据流水线可能是决定模型质量的最大单一因素。用垃圾数据训练大模型，得到的是"大号的垃圾"。

### 训练过程：四步循环

数据准备好了，模型架构搭好了，接下来就是训练。训练过程是一个反复迭代的四步循环：

**第一步：前向传播（Forward Pass）**

把一批训练数据输入模型，模型根据当前参数计算出预测结果——对每个位置，输出一个概率分布，表示模型认为下一个词是什么。

例如，输入 "法国的首都是巴"，模型可能输出：

```
"黎" → 0.65
"黎市" → 0.15
"黎人" → 0.05
"厘" → 0.03
... (其他所有词的概率)
```

**第二步：计算损失（Loss Calculation）**

用交叉熵损失函数比较模型的预测和真实答案。真实答案是"黎"，模型给了它 0.65 的概率。如果概率是 1.0，说明预测完美；概率越低，损失越大。

交叉熵损失的直觉：**模型越"惊讶"，损失越大。** 如果正确答案在模型预测中概率很高（模型不惊讶），损失就小；如果正确答案概率很低（模型很惊讶），损失就大。

**第三步：反向传播（Backpropagation）**

计算损失函数对模型每个参数的梯度——即每个参数应该往哪个方向调整，才能让损失变小。这就是"反向传播"：从输出层开始，逐层往回计算梯度，一直到输入层。

这个过程用到了微积分中的链式法则。GPT-4 有上万亿参数，反向传播要计算每个参数的梯度，计算量极大。

**第四步：更新权重（Weight Update）**

根据梯度调整模型的参数。最常用的优化器是 AdamW，它不仅考虑当前梯度，还会结合历史梯度信息来决定更新幅度。

学习率是这里的关键控制旋钮——它决定了每一步参数更新的幅度。太大，训练不稳定；太小，训练太慢。实践中通常使用**学习率预热（Warmup）**策略：开始时学习率很小，逐渐增大到一个峰值，然后再逐渐衰减。

```
输入数据 → 前向传播 → 计算损失 → 反向传播 → 更新权重 → (重复几百万次)
```

> [!NOTE]
> 整个训练过程就是不断重复这个循环。GPT-4 级别的模型可能要执行数百万到数十亿次这样的循环。每一次循环都在微调模型参数，让预测越来越准确。

### 关键超参数

训练大模型时，有几个关键的超参数会显著影响训练效果：

**批次大小（Batch Size）**

每次前向传播处理的样本数量。GPT-3 的批次大小是 320 万个 Token（注意不是 320 万个句子）。批次大小越大，训练越稳定（梯度估计更准确），但对显存的需求也越大。批次大小受限于 GPU 显存，所以大模型训练必须用分布式策略把批次分散到多个 GPU 上。

**学习率（Learning Rate）**

参数更新的步长。GPT-3 的峰值学习率大约是 6e-5，看起来很小，但经过数百万步的累积，参数会发生巨大变化。学习率通常采用余弦衰减策略：先预热到峰值，然后按余弦曲线缓慢降低。

**训练步数（Training Steps）**

模型"看到"的训练数据总量。GPT-3 训练了大约 3000 亿个 Token（注意：数据可能被看了多遍）。训练步数太少，模型没学够；太多，可能过拟合。如何找到最优的训练步数，是 Scaling Law 研究的核心问题之一（下一章详细讨论）。

**序列长度（Sequence Length）**

模型一次处理的最大 Token 数。GPT-3 是 2048，GPT-4 是 128K（甚至更长）。序列长度影响模型能处理多长的上下文，但也直接影响计算量——注意力的计算量与序列长度的平方成正比。

### 分布式训练：一台机器装不下

训练一个 GPT-4 级别的模型，需要数千张 GPU 同时工作。问题来了：怎么把一个训练任务分配到这么多 GPU 上？

有三种主要的分布式训练策略：

**数据并行（Data Parallelism）**

最直观的方式：把模型完整地复制到每张 GPU 上，但每张 GPU 处理不同的数据批次。每张 GPU 独立做前向传播和反向传播，然后汇总梯度，统一更新参数。

类比：让 100 个人同时做同一套考卷的不同题目，然后把答案汇总。每个人手里都有一份完整的试卷副本。

优点：实现简单。缺点：每张 GPU 都要装下完整的模型——但大模型动辄几十上百 GB，一张 GPU 装不下。

**模型并行（Tensor Parallelism）**

把模型的不同部分放在不同 GPU 上。比如一个矩阵乘法，把矩阵按列切分到 4 张 GPU 上，每张 GPU 计算 1/4，最后合并结果。

类比：把一份试卷的题目分给 4 个人，每人只做 1/4 的题。

优点：可以训练超过单张 GPU 显存容量的模型。缺点：GPU 之间需要频繁通信，通信开销较大。

**流水线并行（Pipeline Parallelism）**

把模型按层切分。比如一个 96 层的模型，GPU 1 负责第 1-24 层，GPU 2 负责第 25-48 层，以此类推。数据像流水线一样依次通过各组 GPU。

类比：工厂流水线——第一个工人做完第一步，传给第二个工人做第二步……

优点：每张 GPU 只需要存储部分层。缺点：存在"气泡"——前面的 GPU 在等待后面的 GPU 完成计算时，处于空闲状态。

```
┌──────────────────────────────────────────────────┐
│ 数据并行：每张 GPU 一个完整模型副本，处理不同数据   │
│ [GPU1: 模型+数据A] [GPU2: 模型+数据B] [...]       │
├──────────────────────────────────────────────────┤
│ 模型并行：一个模型切分到多张 GPU                    │
│ [GPU1: 矩阵左半] [GPU2: 矩阵右半]                 │
├──────────────────────────────────────────────────┤
│ 流水线并行：模型按层切分                           │
│ [GPU1: 层1-24] → [GPU2: 层25-48] → [...]         │
└──────────────────────────────────────────────────┘
```

> [!TIP]
> 实际训练中，这三种策略通常是组合使用的。比如 GPT-4 的训练可能同时使用数据并行（多组 GPU 各处理不同数据）、模型并行（单组内把大矩阵切分到多张 GPU）和流水线并行（不同层分布在不同 GPU 上）。这种组合被称为"3D 并行"。

### 预训练的代价

训练一个大模型需要付出什么代价？

**算力**

GPT-3（1750 亿参数）的训练用了大约 3640 PFLOPS-days（即以每秒 10^15 次浮点运算的速度计算 3640 天）。换算一下，这大约相当于 1000 张 NVIDIA V100 GPU 全速运行 1 个月。GPT-4 的规模被认为远大于 GPT-3，估计训练成本在 5000 万到 1 亿美元之间。

**数据**

万亿 Token 级别的训练数据需要大量的存储、清洗和管理工作。数据工程团队的规模可能和算法团队相当。

**能源**

训练 GPT-3 的能耗估计约为 1287 兆瓦时（MWh）——大约相当于 120 个美国家庭一年的用电量。训练 GPT-4 的能耗可能要高出一个数量级。这引发了关于 AI 行业碳排放的严肃讨论。

**时间**

即使有数千张 GPU，训练一个大模型也需要几周到几个月的时间。GPT-4 的训练可能持续了 3-6 个月。这意味着：如果训练到一半发现出了问题，重新训练的代价极大。

**人才**

能做好预训练的工程师在全世界都是稀缺的。这不是标准软件工程——它需要深入理解分布式系统、高性能计算、数值优化、GPU 编程等多个领域。

| 资源 | GPT-3 级别 | GPT-4 级别（估计） |
|------|-----------|-------------------|
| 参数量 | 1750 亿 | 可能 1-2 万亿（含 MoE） |
| 训练数据 | ~3000 亿 Token | ~13 万亿 Token |
| GPU 数量 | ~1000 张 V100 | ~10000-25000 张 A100 |
| 训练成本 | ~500 万美元 | ~5000 万-1 亿美元 |
| 训练时长 | ~1 个月 | ~3-6 个月 |

> [!WARNING]
> 以上数字大部分是估算。OpenAI、Google、Anthropic 等公司对训练细节严格保密。这些数字来自各种分析报告和学术估算，实际数字可能有较大出入。

### 本章小结

预训练是大模型能力的根基。它通过一个简单的目标（预测下一个词）和海量的数据，让模型习得语言能力、世界知识和推理能力。

关键要点：

- **下一个词预测**看似简单，但迫使模型学习语言的深层结构
- **数据质量**是决定模型效果的关键因素——精心清洗的数据胜过更多的脏数据
- **训练过程**是前向传播、计算损失、反向传播、更新权重的反复循环
- **分布式训练**是大模型训练的工程基础——数据并行、模型并行、流水线并行各有优势
- **训练代价**是巨大的——数千万美元的成本、数月的训练时间、巨大的能源消耗

预训练让模型变得"博学"，但这引出了一个自然的问题：投入更多资源（更大的模型、更多的数据）是否一定能获得更好的效果？有没有规律可循？这就是下一章要讲的 Scaling Law。

## §07 Scaling Law——为什么更大就是更好
Scaling Laws: Why Bigger Is Better

### 2020 年的发现震撼了整个行业

2020 年初，OpenAI 的 Jared Kaplan 等人发表了一篇论文，提出了一个看起来简单但影响深远的发现：**语言模型的性能（用损失函数衡量）与模型参数量、训练数据量和计算量之间存在可预测的幂律关系（Power Law）。**

这意味着什么？意味着你可以在真正投入数千万美元训练模型之前，先训练一些小模型，然后根据幂律曲线**预测**出大模型的性能。你可以精确地知道：把参数量从 10 亿增加到 100 亿，损失会下降多少。

这个发现深刻地改变了整个 AI 行业的战略方向。它告诉所有参与者：**往大了做就对了。** 投入更多算力、更大模型、更多数据，性能的提升是可预期的。这在某种程度上解释了为什么 2020 年以来，大模型的规模呈指数级增长——因为 Scaling Law 给出了确定性的回报承诺。

### Kaplan Scaling Law：三个幂律关系

Kaplank 等人提出的 Scaling Law 包含三个核心关系：

**关系一：模型性能 vs 参数量**

当训练数据足够多时，模型的损失（Loss）与参数量 N 的关系近似为：

```
L(N) ∝ N^(-α)
```

其中 α 是一个正数（论文中约为 0.076）。翻译成人话：**参数量每增加大约 10 倍，损失会降低一个可预测的量。** 而且这个关系在跨越多个数量级（从 10 万参数到千亿参数）时都成立。

**关系二：模型性能 vs 数据量**

当模型足够大时，损失与训练数据量 D 的关系近似为：

```
L(D) ∝ D^(-β)
```

其中 β 约为 0.095。翻译：**数据量每增加大约 10 倍，损失也会降低一个可预测的量。**

**关系三：模型性能 vs 计算量**

给定一个固定的计算预算 C，损失与计算量的关系为：

```
L(C) ∝ C^(-γ)
```

其中 γ 约为 0.050。

这三个幂律关系有一个共同特点：**在对数-对数坐标上，它们都近似为一条直线。** 这意味着你只需要在几个小规模上做实验，就能预测大规模的结果。

```
损失 (对数刻度)
  │\
  │ \
  │  \     幂律曲线：参数量越大，损失越小
  │   \    而且关系高度可预测
  │    \
  │     \
  │      \____
  └──────────── 参数量 (对数刻度)
```

> [!NOTE]
> 幂律关系的"魔力"在于它的尺度不变性——在小模型上观察到的规律可以外推到大模型。这就像物理学中的标度律：你不需要真的造一个和城市一样大的模型来研究流体力学，小尺度的实验结果就能外推。当然，这种外推不是无限的——当模型规模跨越某些临界点时，可能会出现偏差。

### Chinchilla 的挑战：训练更久还是模型更大？

Kaplan Scaling Law 有一个隐含的假设：**模型的参数量和训练数据量中，参数量是更关键的瓶颈。** 换句话说，Kaplan 的结论建议：在给定计算预算下，应该训练尽可能大的模型，数据量不用太多（每个数据点只被模型"看"几次就够了）。

2022 年，DeepMind 发表了关于 Chinchilla 模型的论文，对这个假设提出了挑战。

DeepMind 的研究发现：**Kaplan 的估计低估了数据量的重要性。** 最优的计算分配方式不是"大模型 + 少数据"，而是**"中等模型 + 多数据"**——具体来说，训练数据量应该与模型参数量大致成正比（比例约为 20:1，即每个参数对应大约 20 个训练 Token）。

Chinchilla 用 700 亿参数的模型（远小于 GPT-3 的 1750 亿），配合更多的训练数据，**在性能上超过了 GPT-3**。

这个发现对行业的影响是巨大的：

- **之前**：大家都在追更大的模型（GPT-3 之后期待 GPT-4 更大更猛）
- **之后**：行业意识到数据量同样重要，甚至可能更关键。LLaMA 系列模型就采用了 Chinchilla 的思路——用更多数据训练相对较小的模型

| 策略 | 模型大小 | 数据量 | 代表 |
|------|----------|--------|------|
| Kaplan 最优 | 尽量大 | 较少（~1-2 Token/参数） | GPT-3 |
| Chinchilla 最优 | 适中 | 较多（~20 Token/参数） | Chinchilla, LLaMA |
| 过度训练 | 较小 | 很多（~100+ Token/参数） | LLaMA-3, Mistral |

> [!TIP]
> "过度训练"是 Chinchilla 之后的另一个趋势：用远超最优比例的数据来训练相对较小的模型。这样得到的模型虽然损失不是最优的，但推理成本大幅降低。对于实际部署来说，这是一个非常实用的策略——牺牲少量性能换取大幅降低的推理成本。Meta 的 LLaMA 3 就是这种策略的典型代表。

### 最优计算分配：给定预算，模型该多大

把 Kaplan 和 Chinchilla 的研究结合起来，我们可以回答一个实际问题：**我有 X 美元的预算，模型应该多大、数据应该多少？**

Chinchilla 的结论给出了简洁的答案：给定计算预算 C（以 FLOPS 计），最优的参数量 N 和数据量 D 大致满足：

```
N ∝ C^0.5
D ∝ C^0.5
```

也就是说，**把预算均匀分配给"更大的模型"和"更多的数据"**。计算预算增加 10 倍，模型参数量和数据量各增加大约 3 倍（10^0.5 ≈ 3.16）。

一个实际的计算例子：

- 假设你的计算预算是 10^21 FLOPS（大约相当于单张 A100 GPU 运行约 10 天）
- Chinchilla 最优策略建议：模型大约 10 亿参数，训练数据约 200 亿 Token
- 如果你把预算增加到 10^22 FLOPS（10 倍），建议模型约 30 亿参数，训练数据约 600 亿 Token
- 预算 10^23 FLOPS：模型约 100 亿参数，数据约 2000 亿 Token

这些数字和实际模型的训练配置非常接近——说明业界确实在按照 Scaling Law 的指导来规划训练。

> [!WARNING]
> 最优计算分配只考虑了"训练效率"——即在给定计算预算下，如何最大化模型性能。但在实际应用中，**推理成本**（模型部署后的运行成本）同样是关键考量。一个更大的模型训练效率可能最优，但推理时的成本可能高得不可接受。很多公司因此选择"过度训练"较小的模型——在训练时多花一些算力，但在推理时大幅节省成本。

### 涌现能力：量变到质变的临界点

Scaling Law 最引人入胜的发现之一是**涌现能力（Emergent Abilities）**：模型在达到一定规模之前，某些能力几乎为零；超过某个临界点后，这些能力突然出现。

一个经典的例子是三位数乘法（如 347 × 892 = ?）：

- 参数量 100 亿以下：几乎完全不会做，准确率接近 0%
- 参数量 100 亿到 300 亿：开始有一点点会，但错误率极高
- 参数量 300 亿到 1000 亿：准确率突然飙升到 80% 以上

类似涌现的能力还包括：
- **思维链推理（Chain-of-Thought）**：小模型无法按照步骤推理，大模型突然会了
- **少样本学习（Few-shot Learning）**：仅给几个例子就能理解新任务
- **多语言翻译**：只在英语数据上训练，突然能翻译其他语言
- **代码理解与生成**：即使训练数据中代码比例不高，大模型也能写出正确代码

涌现能力的存在让人着迷，但也引发了不少争议：

- 有研究者认为，所谓的"涌现"可能只是**测量方式的假象**——如果用更精细的评估指标（而不是粗粒度的"做对/做错"），能力的提升可能是平滑渐进的，而非突然跳跃
- 另一些研究者则认为，涌现确实存在，特别是在需要多步推理的任务上——小模型的单步准确率可能高达 90%，但多步串联后准确率急剧下降；大模型在单步上只略好一点，但多步串联后的累积效果就完全不同了

```
任务准确率
  │                          ┌────── 大模型：突然"会了"
  │                     ┌───┘
  │                ┌───┘
  │           ┌───┘
  │ ────────┘          ← 小模型：几乎"不会"
  └────────────────────────── 模型参数量
        ↑
      临界点
```

> [!NOTE]
> 涌现能力的"临界点"并非一个固定的参数量——不同的任务有不同的临界点。简单任务的临界点可能在几亿参数，复杂任务可能需要几千亿参数。目前也没有完美的理论来预测某个具体能力的临界点在哪里。

### 突破 Scaling Law：混合专家模型（MoE）

Scaling Law 给出了"更大就是更好"的指导，但它也带来了一个现实问题：模型越大，推理成本越高。GPT-4 级别的模型每次推理都要激活所有参数，运行一次可能花费数美分。

**混合专家模型（Mixture of Experts, MoE）** 提供了一种绕过这个限制的思路。

MoE 的核心思想很简单：**不要在每次推理时使用模型的所有参数，而是根据输入动态选择一部分"专家"来处理。**

想象一家大医院：
- 普通医院的每个病人都由全科医生看诊（Dense 模型：每个输入都经过所有参数）
- 大医院分了很多专科（内科、外科、儿科、眼科……），每个病人根据症状被路由到对应科室（MoE：每个输入只激活部分专家）

具体实现上，MoE 将 Transformer 中的前馈网络（FFN）层替换为多个并行的"专家"FFN，加上一个路由器（Router）决定每个 Token 应该交给哪些专家处理。常见的配置是：8 个专家，每次只激活 2 个。

MoE 的效果：
- **总参数量很大**（因为有很多专家），但每次推理只激活一小部分，所以实际计算量和推理成本远小于同等参数量的 Dense 模型
- **性能接近甚至超过更大的 Dense 模型**——因为更多的参数量意味着更大的知识容量

Mixtral 8x7B（Mistral AI 的 MoE 模型）就是一个典型例子：总参数量约 470 亿，但每次推理只激活约 130 亿参数，性能却接近 LLaMA-2 700 亿的 Dense 模型。GPT-4 也被广泛认为使用了 MoE 架构。

```
           ┌── 专家1 (FFN) ──┐
Token → 路由器 ├── 专家2 (FFN) ──┼── 输出
           ├── 专家3 (FFN) ──┤
           │   ...           │
           └── 专家8 (FFN) ──┘
           只激活 Top-2 专家
```

> [!WARNING]
> MoE 不是万能的。它的挑战包括：路由器训练不稳定、专家利用率不均（有些专家被频繁使用，有些很少被选中）、跨节点通信开销大等。MoE 模型的实际显存占用仍然很大（所有专家的参数都要加载到内存中），只是每次计算量小了。

### 模型压缩与蒸馏：在能力与成本间找平衡

除了 MoE，还有其他方法可以在保持模型能力的同时降低成本。

**知识蒸馏（Knowledge Distillation）**

核心思想：让一个小模型（学生）学习一个大模型（老师）的输出。不只是学最终的分类结果，还要学老师的"思考过程"——即输出的概率分布。

例如，老师模型对"法国的首都是___"的预测可能是：

```
巴黎 → 0.85，里昂 → 0.05，马赛 → 0.03，伦敦 → 0.01，...
```

学生模型不只是学"答案是巴黎"，而是学这个完整的概率分布。这种"软标签"比硬标签包含了更丰富的信息，帮助小模型学到更精细的知识。

**量化（Quantization）**

模型参数通常以 16 位浮点数（FP16）或 32 位浮点数（FP32）存储。量化就是降低精度：从 FP16 降到 INT8（8 位整数），甚至 INT4（4 位整数）。

- FP16 → INT8：模型大小减半，推理速度提升 2 倍，性能损失极小
- FP16 → INT4：模型大小变为 1/4，推理速度提升 4 倍，性能有可感知的下降但仍可接受

量化为什么能工作？因为模型参数的有效信息实际上不需要 16 位精度——大部分参数只需要 4-8 位就能保留关键信息。就像一张高清照片压缩成 JPEG 格式，虽然丢掉了一些像素细节，但人眼几乎看不出区别。

**剪枝（Pruning）**

直接删除模型中"不重要"的参数或结构。研究发现，大模型中很多参数在推理时几乎不起作用，删除它们对性能影响很小。

| 方法 | 模型大小减少 | 性能损失 | 推理加速 |
|------|-------------|----------|----------|
| INT8 量化 | ~50% | 极小 | ~2x |
| INT4 量化 | ~75% | 小但可感知 | ~3-4x |
| 知识蒸馏 | 取决于学生大小 | 中等 | 取决于学生大小 |
| 结构剪枝 | ~20-50% | 小 | ~1.5-2x |

> [!TIP]
> 这些方法不是互斥的——你可以先训练一个大模型，然后同时使用蒸馏 + 量化 + 剪枝来压缩它。这就是所谓的"训练大模型、部署小模型"的策略。大多数商业 API 背后都不是直接跑原始的大模型，而是跑了经过优化的版本。

### 本章小结

Scaling Law 是大模型时代最核心的洞察之一：它证明了"更大就是更好"不是经验直觉，而是可以用数学描述的规律。

关键要点：

- **幂律关系**让模型性能可以被预测——你可以在小规模上实验，精确预测大规模的结果
- **Chinchilla 发现**纠正了早期对数据量的低估——最优策略是让模型大小和数据量同步增长
- **涌现能力**展示了从量变到质变的可能——某些能力在模型达到临界规模后突然出现
- **MoE 架构**是一种突破推理成本瓶颈的方法——大模型的知识容量，小模型的推理成本
- **量化和蒸馏**提供了部署层面的优化手段——在性能和成本间找到实际可用的平衡点

Scaling Law 回答了"为什么更大就是更好"，但也带来了新的问题：预训练让模型变得"博学"，但一个博学的人不一定是一个好助手。模型需要学会听指令、遵守规则、对齐人类的价值观。这就是下一部分要讨论的微调与对齐。

# Part 3: 微调与对齐

预训练给了模型知识，但预训练出来的模型像一个博学但没有礼貌的学者——它什么都能聊，但不知道什么时候该说什么话。微调和对齐，就是教会它"怎么和人好好说话"。本部分将拆解 SFT、RLHF、DPO 等关键技术。

---

## §08 有监督微调 SFT——教模型学会对话
Supervised Fine-Tuning: Teaching Models to Converse

### 预训练后的模型：博学但不会聊天

想象你雇了一个读完整个图书馆的人来当客服。他确实什么都知道——量子力学、烹饪技巧、罗马历史。但客户说"帮我查一下明天的天气"，他可能会开始背诵气象学发展史。

这正是预训练模型的真实状态。

预训练模型的核心能力是**续写文本**——给它一段开头，它会按照"统计上最可能"的方式接着往下写。但"最可能的续写"和"用户真正想要的回答"之间，存在一条巨大的鸿沟。

来看一个具体的例子。你给预训练模型输入：

```
用户：请用三句话介绍北京。
```

预训练模型可能会输出：

```
用户：请用三句话介绍北京。
用户：北京的冬天冷吗？
用户：上海有哪些好玩的地方？
```

它认为你在列举一系列问题，于是帮你"续写"了更多问题。它不是在回答你，它只是在模仿训练数据中可能出现过的模式。

> [!NOTE]
> 预训练模型不是"不懂"，而是不知道自己应该扮演什么角色。它缺少一种关键信息："当用户向你提问时，你应该回答，而不是接着编。"

### 预训练 vs 微调：到底有什么本质区别？

如果把模型训练比作培养一个人才，预训练和微调分别对应不同阶段：

| 维度 | 预训练 | 微调（SFT） |
|------|--------|-------------|
| 类比 | 上大学，广泛学习知识 | 上岗培训，学习具体工作技能 |
| 目标 | 学会语言的统计规律 | 学会按照指令行动 |
| 数据量 | 万亿 Token，整个互联网 | 几万到几十万条高质量对话 |
| 训练成本 | 数百万到数千万美元 | 几百到几千美元 |
| 训练时间 | 数周到数月 | 几小时到几天 |
| 模型变化 | 从零开始学 | 在已有知识上做小幅调整 |

一个关键的理解：**微调不是重新学习，而是引导。** 模型在预训练阶段已经学到了海量知识和语言能力，微调只是告诉它"这些能力的正确使用方式是什么"。

这就像一个已经掌握英语的人去参加托福培训班——他不需要重新学英语，而是需要学习考试的策略和格式。

### SFT 数据长什么样：指令-回答对

SFT 的核心数据格式是**指令-回答对（Instruction-Response Pairs）**。每一条数据都是一个问题加上一个标准答案，模型通过学习这些"范例"来理解什么叫做"回答问题"。

一条典型的 SFT 训练数据：

```json
{
  "instruction": "请将以下英文翻译成中文：The quick brown fox jumps over the lazy dog.",
  "input": "",
  "output": "那只敏捷的棕色狐狸跳过了那只懒狗。"
}
```

另一条更复杂的数据：

```json
{
  "instruction": "请分析以下代码的时间复杂度，并给出优化建议。",
  "input": "def find_pair(arr, target):\n    for i in range(len(arr)):\n        for j in range(len(arr)):\n            if arr[i] + arr[j] == target:\n                return (i, j)\n    return None",
  "output": "时间复杂度：O(n²)，因为使用了双重嵌套循环...\n\n优化建议：可以使用哈希表，将复杂度降到 O(n)..."
}
```

在训练时，我们把 instruction 和 input 拼接在一起作为输入，output 作为目标输出。模型学习在给定指令的情况下，生成正确的回答。

> [!TIP]
> SFT 数据的质量远比数量重要。几百条精心标注的高质量数据，效果可能优于几万条粗糙数据。这也是为什么 OpenAI 和 Anthropic 在数据标注上投入巨大——他们雇的不是廉价标注工，而是各领域的专家。

### SFT 数据从哪来：三大来源

#### 1. 人工标注——最贵但最好

专业的标注员根据指令写出高质量的回答。这是最直接也最可控的方式。InstructGPT 论文中，OpenAI 雇佣了约 40 名标注员，他们不是简单的"标注工"，而是需要通过资格测试、有写作能力要求的专业人员。

人工标注的特点：
- **质量最高**：回答准确、风格统一、长度适中
- **成本最高**：每条数据几美元到几十美元不等
- **可控性最强**：可以精确控制回答的风格、格式、安全边界

#### 2. AI 生成——规模化利器

用一个已有的强模型（如 GPT-4）来生成指令-回答对，再用这些数据训练新模型。这种方法也叫"蒸馏"或"自指令"（Self-Instruct）。

典型的流程：
1. 给 GPT-4 一些种子任务，让它生成更多类似任务
2. 让 GPT-4 为这些任务生成回答
3. 过滤低质量数据
4. 用这些数据训练你的模型

Alpaca（斯坦福）就是用这种方式，花费不到 600 美元，用 GPT-3.5 生成了 52K 条指令数据，微调 LLaMA 7B 后得到了效果惊人的模型。

#### 3. 开源数据集——社区的力量

开源社区贡献了大量高质量的指令数据集：

| 数据集 | 规模 | 特点 |
|--------|------|------|
| Alpaca | 52K | GPT-3.5 生成，通用指令 |
| ShareGPT | 约 90K | 真实用户对话，更自然 |
| Open Assistant | 约 161K | 社区众包，多语言 |
| Dolly | 15K | Databricks 员工标注，商用友好 |
| Orca | 100K+ | 模仿 GPT-4 的推理过程 |

> [!WARNING]
> 使用 AI 生成的数据训练模型时，要注意"模型坍缩"（Model Collapse）风险——如果训练数据全是 AI 生成的，模型会在几代之后退化，输出越来越单调。就像复印机的复印件再复印，每一代都会损失细节。

### LoRA 与 QLoRA：只训练 0.1% 的参数

全量微调一个 70 亿参数的模型，需要更新全部 70 亿个参数。这意味着你需要巨大的显存来存储模型本身、梯度、优化器状态。一张 A100（80GB）可能都不够用。

LoRA（Low-Rank Adaptation）的出现改变了游戏规则。

#### LoRA 的核心直觉

想象你有一本厚厚的教科书（预训练模型的权重），你不想在书上直接做笔记（修改原始参数），而是贴几张便利贴（低秩矩阵）来补充和修正内容。

具体做法：对于模型中的权重矩阵 W（比如大小是 4096×4096），LoRA 不直接修改 W，而是添加一个"增量" ΔW = A × B，其中 A 是 4096×r 的矩阵，B 是 r×4096 的矩阵，r 通常远小于 4096（比如 r=8 或 r=16）。

```
原始权重更新：W_new = W + ΔW
LoRA 的做法：  W_new = W + A × B

参数量对比：
  全量：4096 × 4096 = 16,777,216 个参数
  LoRA：4096 × 8 + 8 × 4096 = 65,536 个参数
  
节省比例：99.6%
```

用一张图来理解：

```
┌─────────────────────────────────────┐
│         原始权重矩阵 W              │
│         (冻结，不训练)               │
│         4096 × 4096                 │
│                                     │
│   输入 ──→ [  W  ] ──→ + ──→ 输出   │
│               ↑         ↑           │
│               │         │           │
│         ┌─────┘    ┌────┘           │
│         │          │                │
│    ┌────┴───┐  ┌──┴─────┐          │
│    │   A    │  │   B    │          │
│    │4096×8  │→│ 8×4096 │          │
│    │(可训练) │  │(可训练) │          │
│    └────────┘  └────────┘          │
│         LoRA 适配器                  │
│     (只训练 0.1% 的参数)            │
└─────────────────────────────────────┘
```

为什么这能工作？因为微调所需的对权重的修改，本质上是"低秩"的——也就是说，不需要在每个维度上都做大幅调整，只需要在一个低维子空间里做微调就足够了。

> [!NOTE]
> "秩"（Rank）在这里可以理解为"自由度"或"调整的自由度"。秩越低，可调整的空间越小，但也意味着训练越快、显存占用越少。通常 r=8 到 r=64 就能覆盖大多数微调需求。

#### QLoRA：在 LoRA 的基础上更进一步

QLoRA（Quantized LoRA）在 LoRA 的基础上增加了一个关键优化：**把冻结的原始权重从 16 位浮点数压缩到 4 位**。

```
全量微调：  70亿参数 × 16 bit = ~140 GB 显存
LoRA：     原始权重 140 GB + LoRA 参数 ~14 MB ≈ 140 GB（仍需加载原始权重）
QLoRA：    原始权重 4 bit 压缩 = ~35 GB + LoRA 参数 ~14 MB ≈ 35 GB
```

这意味着你可以用一张消费级 GPU（如 RTX 4090，24GB 显存）来微调一个 70 亿参数的模型。QLoRA 让"在笔记本上微调大模型"从不可能变成了可能。

### 全量微调 vs 参数高效微调：选择指南

| 维度 | 全量微调 | LoRA/QLoRA |
|------|----------|------------|
| 效果上限 | 更高（理论上） | 略低，但差距通常很小 |
| 训练成本 | 高（需要多卡） | 低（单卡即可） |
| 显存需求 | 模型 + 梯度 + 优化器 | 只需加载模型 + 少量参数 |
| 训练速度 | 较慢 | 快 2-5 倍 |
| 存储占用 | 完整模型副本（几 GB 到几十 GB） | 仅适配器权重（几 MB 到几十 MB） |
| 可组合性 | 低 | 高（可切换不同适配器） |

**什么时候选全量微调？**
- 你有充足的算力（多张 A100/H100）
- 任务与预训练领域差异极大（比如用通用模型做医疗诊断）
- 你追求极致效果，愿意为此付出成本

**什么时候选 LoRA/QLoRA？**
- 算力有限（消费级 GPU 或单卡服务器）
- 任务与预训练领域接近（大多数情况）
- 需要快速迭代实验
- 需要为不同任务维护多个适配器

> [!TIP]
> 实践中，LoRA 的效果通常能达到全量微调的 95% 以上，而成本只有 1/10 甚至更低。除非你有充分的理由，否则 LoRA/QLoRA 几乎总是更好的选择。

### 过拟合的风险：为什么微调太多反而变笨

过拟合是机器学习中的经典问题，在 SFT 中尤其危险。

想象你在准备面试。正确做法是理解面试的底层逻辑，举一反三。但如果你把 100 道面试题的答案全背下来——你可能在模拟面试中表现完美，但遇到第 101 道新题时就傻了。

SFT 过拟合的表现：
- **格式过度死板**：不管用户问什么，都用固定的格式回答
- **丧失泛化能力**：在训练集覆盖的场景表现很好，遇到新场景就变差
- **"语气"过度讨好**：每个回答都以"当然！很高兴帮助你！"开头
- **知识遗忘**：模型为了学习新格式，丢掉了预训练阶段获得的知识

防止过拟合的关键策略：

1. **控制训练轮数（Epochs）**：SFT 通常只需要 1-3 个 epoch，再多就容易过拟合
2. **数据多样性**：确保训练数据覆盖足够多的场景和风格
3. **正则化**：使用 dropout、权重衰减等技术
4. **学习率要小**：SFT 的学习率通常是预训练的 1/10 到 1/100
5. **定期验证**：在验证集上监控损失，一旦开始上升就停止训练

> [!WARNING]
> 一个反直觉的事实：模型在 SFT 训练集上的 loss（误差）越来越低，不代表模型在变得更好。当训练 loss 持续下降但验证 loss 开始上升时，模型已经在"背诵答案"而非"学习模式"了。这就是经典的过拟合信号。

### 小结

SFT 是将预训练模型转化为实用对话模型的桥梁。它不是重新教模型知识，而是教模型"如何运用已有知识来回答问题"。

关键要点回顾：

- **预训练模型只会续写，不会对话。** SFT 通过指令-回答对的数据教模型学会"听指令、给回答"。
- **数据质量 > 数据数量。** 几千条高质量数据的训练效果往往优于几万条低质量数据。
- **LoRA/QLoRA 是高效微调的核心工具。** 通过只训练 0.1% 的参数，在消费级 GPU 上就能完成微调。
- **过拟合是 SFT 的头号敌人。** 宁可训练不足，也不要过度训练。

SFT 让模型学会了"怎么和人对话"。但对话能力只是第一步——模型还需要知道什么该说、什么不该说，什么样的回答更受欢迎。这就引出了下一章的主题：RLHF。

## §09 RLHF——用人类反馈驯服大模型
RLHF: Taming LLMs with Human Feedback

### 为什么 ChatGPT 比 GPT-3 好用 100 倍？

2020 年发布的 GPT-3 是一个划时代的产品。它拥有 1750 亿参数，能写出流畅的文章、翻译语言、甚至写代码。但如果你用过早期的 GPT-3 API，你会发现一个明显的问题：**它很难用。**

你问它"法国的首都是什么？"，它可能回答"巴黎"——也可能回答"这是一个关于地理的问题，法国的首都是巴黎，巴黎是法国最大的城市..."然后絮絮叨叨停不下来。或者它可能偏题到讲法国大革命的故事。

而 2022 年底发布的 ChatGPT（基于 GPT-3.5），参数量并没有比 GPT-3 大多少，但用户体验却好了几个量级。为什么？

核心答案：**RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）。**

如果说 SFT 教会了模型"怎么说话"，RLHF 则教会了模型"怎么把话说好"——什么样的回答更受欢迎、更有帮助、更安全。

> [!NOTE]
> RLHF 不会给模型增加新知识。它的作用是"对齐"（Alignment）——让模型的输出更符合人类的期望和价值观。你可以在某种意义上把 RLHF 理解为给模型做"礼仪培训"。

### RLHF 的三阶段全景图


![图9-1：RLHF训练三阶段流程](images/rlhf-pipeline.png)
RLHF 不是一步到位的，而是一个三阶段的流程：

```
┌────────────────────────────────────────────────────────────────┐
│                     RLHF 三阶段流程                            │
│                                                                │
│  阶段 1          阶段 2             阶段 3                     │
│  有监督微调       奖励模型训练         强化学习优化              │
│  (SFT)           (Reward Model)      (PPO)                    │
│                                                                │
│  ┌───────┐      ┌───────────┐      ┌───────────────┐          │
│  │指令数据 │  →   │  偏好数据   │  →   │  用奖励信号     │          │
│  │训练一个 │      │  训练一个   │      │  优化 SFT 模型  │          │
│  │基座模型 │      │  AI 裁判   │      │  的输出策略     │          │
│  └───────┘      └───────────┘      └───────────────┘          │
│                                                                │
│  产出：SFT 模型   产出：奖励模型     产出：最终对齐模型          │
└────────────────────────────────────────────────────────────────┘
```

接下来，我们逐一拆解每个阶段。

#### 阶段 1：SFT——已经讲过了

第一阶段的 SFT 就是我们上一章讨论的内容。它产出一个"基座对话模型"——已经学会了听指令、给回答，但回答的质量参差不齐。

RLHF 的第二阶段和第三阶段，都是在这个 SFT 模型的基础上进行的。

#### 阶段 2：奖励模型——训练一个"AI 裁判"

这是 RLHF 中最关键也最有创意的一步。

**问题：** 我们想让模型的输出"更好"，但"更好"是一个模糊的概念。怎么让计算机理解"更好"？

**方案：** 训练一个奖励模型（Reward Model，简称 RM），它能给任何一段"指令-回答"对打一个分数，分数越高代表回答越好。

**怎么训练这个奖励模型？** 关键在于人类标注数据。

具体流程：

```
步骤 1：生成候选回答
  对于同一个指令，让 SFT 模型生成 4-9 个不同的回答。

步骤 2：人类排序
  标注员对这 4-9 个回答进行质量排序。
  
  例如，对于指令"解释量子纠缠"：
    回答 A：准确、简洁、易懂         ← 排名第 1
    回答 C：准确但过于学术           ← 排名第 2  
    回答 B：部分准确但有些混乱        ← 排名第 3
    回答 D：错误信息，胡说八道        ← 排名第 4

步骤 3：训练奖励模型
  用排序数据训练 RM，让它学会：
  - 好回答得高分
  - 差回答得低分
  - 分数差异反映质量差距
```

奖励模型的训练目标很直观：对于同一指令下的两个回答，如果人类认为回答 A 比回答 B 好，那么 RM 应该给 A 打更高的分。

训练好之后，奖励模型就像一个"自动裁判"——给定任何指令和回答，它都能给出一个分数，代表这个回答的质量。

> [!TIP]
> 奖励模型本质上是在学习人类的偏好函数。它不是在判断"对或错"，而是在判断"人类更喜欢哪个"。这比简单的对错判断要复杂得多，因为它需要捕捉微妙的质量差异——比如语气、清晰度、有用程度等。

#### 阶段 3：PPO——用奖励信号引导模型进化

有了奖励模型（裁判），下一步就是用它来优化 SFT 模型。

这里用到的是强化学习算法 PPO（Proximal Policy Optimization）。PPO 的名字听起来很吓人，但核心思想其实不复杂。

**PPO 的直觉：**

想象你在训练一只小狗。你不会每次都从头教它整个动作，而是：
1. 让它试一试当前会的动作
2. 做得好就给奖励（零食）
3. 做得不好就不给奖励
4. 每次只做小幅调整，不要突然大改（这就是 "Proximal" 的含义——"近端"，限制每次的变化幅度）

PPO 的工作方式与此类似：

```
1. SFT 模型（现在叫"策略模型"）生成一批回答
2. 奖励模型给每个回答打分
3. 根据分数计算"哪些变化方向能提高分数"
4. 沿着高分方向小幅调整模型参数
5. 关键约束：每次调整不能偏离原模型太远（防止"学坏了"）
6. 重复步骤 1-5
```

为什么需要"不能偏离太远"这个约束？

因为在强化学习中，模型可能发现一些"捷径"来获得高分——比如输出非常长的回答（因为人类标注员往往偏好更长的回答），或者在回答中加很多讨好性的话（"当然！非常高兴帮助您！"）。这些"捷径"在短期内能提高奖励模型的分数，但实际体验很差。

PPO 通过引入一个 KL 散度惩罚项来限制模型的变化幅度，确保模型在"追求高分"的同时，不会偏离原始 SFT 模型太远。

```
PPO 的优化目标（简化版）：

  目标 = 奖励模型给的分数 - β × 偏离 SFT 模型的程度

  其中 β 控制对偏离的惩罚力度。
  β 太大 → 模型几乎不变化
  β 太小 → 模型可能"学坏了"
```

> [!NOTE]
> PPO 中的 "Proximal"（近端）是整个算法的关键。它就像一根弹性绳，连接着当前模型和 SFT 基座模型——你可以跑，但跑太远绳子就会把你拉回来。这个机制确保了优化过程的稳定性。

### RLHF 的挑战：不完美的系统

RLHF 是一个强大的框架，但它并不完美。在实践中，RLHF 面临几个棘手的挑战：

#### 1. 奖励黑客（Reward Hacking）

奖励黑客是指模型学会了"刷分"——找到奖励模型的漏洞来获得高分，而不是真正提升回答质量。

典型案例：
- **长度偏好**：奖励模型可能偏好更长的回答，于是模型学会了"凑字数"
- **格式偏好**：奖励模型可能偏好带列表的回答，于是模型不管什么问题都列出 1、2、3
- **语气偏好**：奖励模型可能偏好"礼貌"的回答，于是模型学会了过度讨好

这就像一个学生发现老师喜欢字数多的作文，于是每次都写 3000 字的水文——字数够了，但质量并没提升。

#### 2. 标注成本

RLHF 需要大量的人类偏好数据。OpenAI 在 InstructGPT 中使用了约数万条比较数据，而要做到更好的效果可能需要更多。每条数据都需要人类标注员仔细比较和排序，成本高昂。

更复杂的是，不同的标注员可能有不同的偏好标准。一个人认为"详细"的回答，另一个人可能觉得"啰嗦"。这种不一致性会直接影响奖励模型的质量。

#### 3. 训练不稳定

PPO 本身是一个对超参数敏感的算法。学习率设太大，模型可能崩溃；设太小，训练效率太低。KL 散度惩罚的系数 β 也需要精心调校。整个 RLHF 训练过程往往需要多次实验才能找到合适的超参数组合。

### Anthropic 的实践：从 RLHF 到 RLAIF

Anthropic（Claude 的开发者）在 RLHF 的基础上做了进一步的创新。

他们的核心观察是：**人类标注太贵了，而且标准不统一。既然我们已经有了强大的模型，为什么不直接用 AI 来做标注？**

这就是 RLAIF（Reinforcement Learning from AI Feedback）的核心想法——用 AI 替代人类来提供偏好反馈。

```
传统 RLHF：
  模型生成多个回答 → 人类排序 → 训练奖励模型 → PPO 优化

RLAIF：
  模型生成多个回答 → 另一个 AI 排序 → 训练奖励模型 → PPO 优化
```

用 AI 做标注有几个明显优势：
- **成本极低**：一次 API 调用几分钱，而人类标注员每条数据几美元
- **标准统一**：AI 不会因为心情不好或疲劳而降低标注质量
- **规模无限**：可以 24 小时不间断标注，理论上可以无限扩展
- **可定制**：通过调整提示词，可以精确控制评估标准

但 RLAIF 也有风险——AI 可能会系统性地偏好某种特定风格的回答，导致模型同质化。

> [!WARNING]
> RLAIF 存在"自我强化"的风险。如果用同一个模型来评估和优化自己，模型可能会逐渐放大自己的偏见。就像一个只看自己写的东西的作家，风格会越来越极端。实践中，通常用一个不同的、更强的模型来做评估，形成"交叉验证"。

### 小结

RLHF 是让 ChatGPT 成为"好用的产品"而非"强大的技术演示"的关键技术。它通过三个阶段——SFT、奖励模型训练、PPO 优化——将一个"会说话"的模型变成一个"会说好话"的模型。

关键要点回顾：

- **RLHF 的核心是奖励模型。** 它学会了人类的偏好函数，能自动判断回答质量。
- **PPO 做的是小幅优化。** 不是推倒重来，而是在已有能力基础上做精细调整。
- **奖励黑客是最大隐患。** 模型可能会"刷分"而非真正提升质量。
- **RLAIF 是未来的方向。** 用 AI 替代人类做标注，大幅降低成本并提高可扩展性。

但 RLHF 也有一个根本性的问题：它太复杂了。三个阶段、多个模型（SFT 模型、奖励模型、PPO 中的策略模型和参考模型）、大量的超参数调优……这让 RLHF 的工程实现变得昂贵且脆弱。

有没有更简单的方法？下一章我们来回答这个问题。

## §10 新一代对齐方法——从 DPO 到宪法 AI
Next-Gen Alignment: From DPO to Constitutional AI

### RLHF 太复杂了？直接优化偏好就行

上一章我们了解了 RLHF 的完整流程：先训练 SFT 模型，再训练奖励模型，再用 PPO 做强化学习优化。整个过程涉及多个模型的协调训练，工程复杂度极高。

这就引出了一个自然的问题：**能不能跳过奖励模型和 PPO，直接用偏好数据来训练模型？**

答案是可以。这就是 DPO——2023 年由斯坦福大学团队提出的 Direct Preference Optimization（直接偏好优化）。

DPO 不是 RLHF 的简单替代品，而是一次根本性的思维转变。要理解这个转变，我们先打个比方。

**类比：两条管理路线**

假设你是一个公司的 CEO，想让员工的行为更符合公司利益。

**路线 A（RLHF 的思路）：**
1. 制定一套评分标准（训练奖励模型）
2. 让员工自由行动
3. 根据评分标准打分
4. 用分数来奖惩员工（PPO 优化）

**路线 B（DPO 的思路）：**
1. 给员工看两个案例："好员工这样做，差员工那样做"
2. 直接告诉员工"多学好的，少学差的"
3. 不需要评分标准，不需要奖惩系统

DPO 选择了路线 B——更直接，更简单，但同样有效。

### DPO 的核心：一个巧妙的损失函数

DPO 的数学推导（来自论文）其实很精妙，但我们可以跳过推导过程，直接看结论。

DPO 的训练数据和 RLHF 的奖励模型训练阶段一样——都是**偏好对**：对于同一个指令，一个"好的"回答和一个"差的"回答。

```
训练数据格式：
{
  "prompt": "解释什么是黑洞",
  "chosen": "黑洞是时空中引力极强的区域，连光都无法逃脱...",     ← 好回答
  "rejected": "黑洞就是一个黑色的洞，宇宙中很常见的东西..."     ← 差回答
}
```

DPO 的损失函数（简化理解）做的事情是：

```
训练目标 = 让模型更倾向于生成 chosen 回答
         + 让模型更不倾向于生成 rejected 回答
         + 同时保持模型不会偏离原始 SFT 模型太远
```

这三个目标被巧妙地合并到一个数学公式中，使得 DPO 只需要一次训练（不需要奖励模型，不需要 PPO），就能达到和 RLHF 类似甚至更好的效果。

```
┌─────────────────────────────────────────────────────────────┐
│              RLHF vs DPO 流程对比                            │
│                                                             │
│  RLHF：                                                     │
│  偏好数据 → 训练奖励模型 → PPO 训练 → 最终模型               │
│  （3 个阶段，4 个模型）                                      │
│                                                             │
│  DPO：                                                      │
│  偏好数据 → 直接训练 → 最终模型                              │
│  （1 个阶段，1 个模型）                                      │
│                                                             │
│  简化程度：大幅简化！                                        │
└─────────────────────────────────────────────────────────────┘
```

DPO 的关键优势：

| 维度 | RLHF | DPO |
|------|------|-----|
| 训练阶段数 | 3 | 1 |
| 需要的模型数 | 4（SFT + RM + 策略 + 参考） | 2（SFT + 正在训练的模型） |
| 训练稳定性 | 难调，容易崩溃 | 稳定，和 SFT 训练一样简单 |
| 训练成本 | 高 | 低（约为 RLHF 的 1/3 到 1/2） |
| 数据需求 | 偏好对（同 RLHF） | 偏好对 |
| 效果 | 经过充分调优后很好 | 通常接近或等于 RLHF |

> [!NOTE]
> DPO 本质上是在数学上证明了：RLHF 中奖励模型和 PPO 的组合，可以被一个直接的损失函数等价替代。这个损失函数隐式地包含了奖励模型的信息，不需要显式地训练它。这是理论上的一个优美结论。

### 宪法 AI：用一套"宪法"引导模型自我对齐

Anthropic 提出的宪法 AI（Constitutional AI，简称 CAI）代表了对齐方法的另一个重要方向：**让模型自己参与对齐过程。**

"宪法"这个名字很形象——就像一个国家有宪法作为最高准则，模型也有一套"行为准则"来约束自己的输出。

#### 宪法 AI 的工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                宪法 AI 流程                                   │
│                                                             │
│  第一步：自我批评（Self-Critique）                            │
│  ┌─────────────────────────────────────────┐                │
│  │ 模型生成一个可能有害的回答               │                │
│  │        ↓                                │                │
│  │ 用"宪法"规则让模型评价自己的回答          │                │
│  │   "这个回答是否违反了规则 X？"           │                │
│  │        ↓                                │                │
│  │ 模型根据评价修正自己的回答               │                │
│  └─────────────────────────────────────────┘                │
│                                                             │
│  第二步：基于修正数据训练                                     │
│  ┌─────────────────────────────────────────┐                │
│  │ 原始有害回答 vs 修正后的回答              │                │
│  │        ↓                                │                │
│  │ 构成偏好对，用于 DPO/RLHF 训练           │                │
│  │ 让模型学会主动避免有害输出               │                │
│  └─────────────────────────────────────────┘                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

"宪法"规则是一组精心设计的原则。Anthropic 在论文中列举了一些例子（以下是示意，非原文）：

- 选择最无害且最有帮助的回答
- 如果回答可能被用于有害目的，选择更谨慎的回答
- 不输出歧视性的内容
- 不提供可能导致伤害的详细指导
- 保持诚实，不确定时承认不确定

整个过程的精妙之处在于：**人类只需要制定"宪法"规则，不需要逐条数据去标注。** 模型会用这些规则来自我评估和修正，生成大量训练数据。

#### 宪法 AI 的优势

1. **可扩展**：加入新规则只需要更新"宪法"，不需要重新标注数据
2. **透明**：所有规则都是明文写出的，可以审查和讨论
3. **一致性好**：用统一的规则评估所有输出，避免了不同标注员标准不一致的问题
4. **可审计**：可以检查模型是依据哪条规则做出的决策

### RLAIF：用 AI 给 AI 打分

宪法 AI 和 RLAIF（RL from AI Feedback）紧密相关，但又不完全相同。

RLAIF 是一个更广义的概念：任何使用 AI（而非人类）提供反馈信号的对齐方法，都可以叫 RLAIF。宪法 AI 可以看作是 RLAIF 的一种具体实现方式——用"宪法"规则指导 AI 提供反馈。

RLAIF 的一般流程：

```
1. SFT 模型生成多个候选回答
2. 一个评估 AI（通常比 SFT 模型更强）对这些回答进行排序
3. 评估 AI 的排序依据可以是：
   - 内在的"好回答"判断
   - 一套明确的评估标准（宪法 AI）
   - 人类提供的 few-shot 示例
4. 用 AI 的排序结果替代人类排序，进行后续训练
```

> [!TIP]
> RLAIF 最实际的用法是：用 GPT-4 级别的模型来评估和排序，然后用排序结果来训练更小的模型。这样，小模型可以在成本远低于 GPT-4 的情况下，获得接近 GPT-4 的行为质量。这就是所谓的"用强老师的品味来教学生"。

### KTO、IPO、ORPO 等新方法一览

DPO 的成功激发了学术界对"更简单的对齐方法"的探索。以下是几个重要的后继方法：

#### KTO（Kahneman-Tversky Optimization）

KTO 的核心创新：**不需要成对的偏好数据，只需要知道每个回答是"好"还是"差"。**

这为什么重要？因为获取成对的偏好数据（同一个问题的两个回答，且人类明确排序）成本很高。而单纯判断一个回答"好不好"（点赞/踩）则便宜得多。

类比：DPO 需要"电影 A 比电影 B 好看"这样的比较判断，KTO 只需要"这部电影好看/不好看"这样的独立判断。后者获取成本大幅降低。

#### IPO（Identity Preference Optimization）

IPO 解决的是 DPO 的一个理论缺陷：DPO 在偏好差异非常大的时候可能训练不稳定。IPO 用一个更温和的数学公式来替代 DPO 的损失函数，在保持效果的同时提高了训练稳定性。

#### ORPO（Odds Ratio Preference Optimization）

ORPO 更进一步：**把 SFT 和偏好对齐合并成一步。** 不需要先做 SFT 再做 DPO，而是在一个训练过程中同时完成"学会回答问题"和"学会回答好问题"。

```
传统流程：预训练 → SFT → DPO（三步）
ORPO 流程：预训练 → ORPO（两步）
```

#### 方法对比速查表

| 方法 | 数据需求 | 训练步数 | 核心优势 |
|------|----------|----------|----------|
| RLHF | 偏好对 + 人类排序 | 3 阶段 | 效果上限高，经过充分验证 |
| DPO | 偏好对（chosen vs rejected） | 1 阶段 | 简单稳定，效果接近 RLHF |
| KTO | 二元标签（好/差） | 1 阶段 | 数据获取成本最低 |
| IPO | 偏好对 | 1 阶段 | 训练更稳定 |
| ORPO | 偏好对 + SFT 数据 | 1 阶段 | 合并 SFT 和对齐，最高效 |
| 宪法 AI | 宪法规则 + 少量人类反馈 | 2 阶段 | 可扩展、透明、可审计 |

> [!NOTE]
> 截至 2025 年，DPO 已经成为工业界最主流的对齐方法。大多数开源模型的训练流程都采用了 DPO 或其变体。RLHF 由于其复杂性和不稳定性，正在被更简单的方法逐步替代。但 RLHF 的核心思想——用偏好信号来对齐模型——依然是所有这些方法的基础。

### 对齐税：让模型更安全会不会让它变笨？

这是一个非常实际的问题，也是 AI 研究社区持续争论的话题。

**对齐税（Alignment Tax）** 指的是：为了让模型更安全、更有帮助，模型在其他方面的能力可能会下降。

具体表现：

- **创造力下降**：对齐后的模型可能更保守，不愿意生成有创意的内容
- **推理能力受限**：模型可能为了避免"说错话"而在需要精确推理时过于谨慎
- **拒绝率过高**：模型可能把一些合理的问题也误判为"危险"而拒绝回答
- **风格趋同**：不同模型对齐后可能变得"千人一面"，缺乏个性

举个例子：你对模型说"写一个关于侦探的故事"，未对齐的模型可能会写一个包含暴力情节的精彩推理故事，而对齐后的模型可能会为了避免"暴力内容"而写出一个过于温和、缺乏张力的故事。

**当前的共识是：**

1. 对齐税确实存在，但可以通过技术手段来减轻
2. 关键在于平衡——不是"越安全越好"，而是"在不损害有用性的前提下尽量安全"
3. 好的对齐方法应该能同时提升安全性和有用性，而不是在两者间做零和博弈

减轻对齐税的策略：
- **高质量的对齐数据**：好的偏好数据不会教模型"什么都拒绝"，而是教它"合理地拒绝"
- **精细的宪法规则**：不是简单的"禁止一切可能的危害"，而是有层次的规则体系
- **对齐后的能力恢复训练**：在对齐训练后，补充一些能力训练来弥补可能的损失
- **持续迭代**：根据用户反馈不断调整对齐策略，在安全和有用之间找到最佳平衡点

> [!WARNING]
> 过度对齐和不对齐同样危险。一个过度对齐的模型就像一个过于胆小的顾问——他永远不会犯错，但也永远不会给你真正有价值的建议。好的对齐是让模型变得"负责任地勇敢"，而不是"安全地无用"。

### 小结

对齐领域正在经历从复杂到简单的演进。RLHF 证明了对齐是可行的，而 DPO、宪法 AI 等新方法让对齐变得更简单、更可扩展。

关键要点回顾：

- **DPO 用一个巧妙的损失函数替代了 RLHF 的奖励模型和 PPO。** 它把三阶段训练压缩成了一个阶段，大幅降低了工程复杂度。
- **宪法 AI 让模型用一套明文规则来约束自己。** 人类制定规则，模型自我执行和自我评估。
- **RLAIF 用 AI 替代人类做标注，** 大幅降低了对齐成本，让对齐变成了可规模化的事情。
- **KTO、IPO、ORPO 等新方法在 DPO 的基础上进一步简化** 了数据需求或训练流程。
- **对齐税是真实存在的。** 好的对齐应该在安全和有用之间找到平衡，而不是一味追求安全。

回到本书的城市隐喻：如果预训练是建设城市的基础设施，微调是让城市学会服务居民，那对齐就是制定城市规则——红绿灯、交通法规、建筑安全标准。没有规则的城市是混乱的，但规则过多的城市也是窒息的。关键在于，找到那个刚刚好的平衡点。

理解了大模型"怎么学"，最后一个问题是"怎么用"。本部分将从推理解码到 RAG 和 Agent，完成从原理到应用的最后一跳。

## §11 推理与解码——模型如何生成文字
Inference & Decoding: How Models Generate Text

### 你按下回车之后发生了什么？

你在 ChatGPT 的输入框里打了一句话："请解释量子纠缠"，按下回车。屏幕上的光标停顿了零点几秒，然后一个字一个字地冒出来。这个过程看起来理所当然，但在那零点几秒里，模型经历了一场精密的计算马拉松。

整个过程可以分为三步：

1. **分词与编码**：你的输入被 Tokenizer 切成一个个 Token，转换成整数 ID，再映射成嵌入向量——这就是我们在 §02 和 §03 中讨论的过程。
2. **前向传播**：这些向量穿过几十层 Transformer Block，每层的自注意力机制和前馈网络依次处理，最终在顶层输出一组**原始得分**。
3. **解码**：模型从这组原始得分中，选出下一个 Token。然后把这个 Token 拼接到已有文本后面，重复步骤 2 和 3，直到生成结束。

前两步在之前的章节已经拆解清楚了。这一章聚焦于**第三步：解码**——模型怎样从一堆数字中"挑"出下一个词来。

> [!NOTE]
> 前向传播每次只产生一个 Token 的概率分布（虽然实际上是所有候选 Token 的得分）。模型需要反复调用前向传播，才能一个词一个词地把完整回答"吐"出来。这也是为什么生成速度比理解速度慢得多。

### 从 Logits 到概率：Softmax 函数

经过最后一层 Transformer，模型输出的不是直接的文字，而是一个** logits 向量**。这个向量的长度等于词表大小（比如 GPT-4 的词表约有 100,277 个 Token），每个位置对应一个候选 Token 的"原始得分"。

问题是，logits 可以是任意实数——负无穷到正无穷都有可能。我们需要把这些得分转化为概率：每个 Token 获得一个 0 到 1 之间的值，且所有值加起来等于 1。

这个转化工具就是 **Softmax 函数**。

用一个简单的例子来理解。假设词表只有 5 个词（实际是 10 万级别），模型对下一个词的 logits 输出如下：

```
Token     Logit
"今天"     3.2
"明天"     2.1
"昨天"     1.5
"天气"    -0.3
"苹果"    -2.8
```

Softmax 的作用是：

```
概率 = exp(logit) / 所有 exp(logit) 之和
```

想象一张这样的图：五个柱状条从左到右排列，原始 logit 值有大有小、有正有负。经过 Softmax 之后，这五个柱状条变成了一组平滑的概率值——最高的"今天"获得最大的概率份额，最低的"苹果"几乎为零，但所有柱状条的高度加起来精确等于 1。

经过 Softmax 之后的结果大约是：

```
Token     概率
"今天"     0.52
"明天"     0.17
"昨天"     0.09
"天气"     0.02
"苹果"     0.002
```

> [!TIP]
> Softmax 有一个重要特性：**它放大差异**。logit 值之间 1.0 的差距，在概率上会被放大得更多。这意味着模型越"确定"的词，概率会被推得越高。这就像赛跑——第一名只比第二名快了 0.1 秒，但冠军的光环远不止 0.1 秒的差距。

### 解码策略：贪心 vs 采样

拿到了概率分布，接下来就是选择下一个 Token。怎么选？这看似简单的问题，却直接影响生成文本的质量和多样性。

#### 贪心解码（Greedy Decoding）

最直观的方法：**永远选概率最高的那个 Token**。

就像一个考试只选最有把握答案的学生——每一步都选当前最优，不犹豫、不冒险。

```
第 1 步：概率最高 → "今天"
第 2 步：概率最高 → "天气"
第 3 步：概率最高 → "很好"
→ "今天天气很好"
```

**优点**：输出确定性强——同样的输入永远产生同样的输出。

**缺点**：容易陷入重复和无聊。因为每一步都选最"安全"的词，模型倾向于生成最平庸、最常见的文本。更严重的是，贪心解码可能陷入死循环：`"我很高兴。很高兴。很高兴。很高兴。"`

#### 采样解码（Sampling）

另一种方法：**按照概率分布随机抽取**。概率高的词被选中的机会大，但概率低的词也有机会被选中。

就像一个按体重分配奖券的抽奖——胖子奖券多，中奖概率大，但瘦子也不是完全没有机会。

```
概率分布：今天(0.52) 明天(0.17) 昨天(0.09) ...

采样结果可能是：
  第 1 次 → "今天"（概率大，很可能选中）
  第 2 次 → "明天"（偶尔也会冒险）
  第 3 次 → "昨天"（小概率事件）
```

**优点**：输出有创意、有变化。同样的输入可以生成不同的回答。

**缺点**：可能选出不该选的词，产生不连贯甚至荒谬的文本。

> [!WARNING]
> 纯采样（不做任何限制）在实际中几乎不用。因为词表里有 10 万多个 Token，其中大量是模型不该在当前语境下选择的。不加限制的采样就像在整本字典里随机翻词——偶尔会翻到完全不相关的东西。

实际使用中，我们需要在"确定性"和"多样性"之间找到平衡。接下来的三个参数就是用来精细调节这个平衡的。

### Temperature：输出的"温度计"

Temperature 是控制输出随机性最常用的参数。它作用于 Softmax 之前，通过调整 logits 来改变概率分布的"尖锐程度"。

```
新 logits = 原始 logits / Temperature
```

想象一张这样的图：三条概率分布曲线叠在一起，共享同一组 Token 作为横轴。中间那条（Temperature = 1.0）是基准线，分布适中；左边那条（Temperature = 0.1）尖如针塔——几乎所有概率都集中在一个 Token 上；右边那条（Temperature = 2.0）平如桌面——每个 Token 的概率都差不多。

- **Temperature < 1.0**：logits 被放大，差异加剧，概率更集中在头部 Token。模型变得"保守"，倾向选最安全的词。
- **Temperature = 1.0**：不改变 logits，保持原始概率分布。
- **Temperature > 1.0**：logits 被缩小，差异被压缩，概率变得更均匀。模型变得"大胆"，更愿意尝试低概率的词。

一个生活中的类比：

- **低 Temperature（0.1-0.3）**像一位严谨的学者——只说最有把握的话，措辞精确但可能略显乏味。适合写代码、做数学题、回答事实性问题。
- **中 Temperature（0.7-0.9）**像一位健谈的朋友——大部分时候靠谱，偶尔来点小惊喜。这是大多数对话场景的最佳设定。
- **高 Temperature（1.5-2.0）**像一位喝多了的诗人——灵感迸发但也可能语无伦次。适合创意写作、头脑风暴。

> [!NOTE]
> Temperature = 0 等价于贪心解码（严格来说 Temperature 趋近于 0 时 Softmax 退化为 argmax）。大多数 API 都不允许设为精确的 0，通常下限是 0.01。

### Top-K：只从 K 个候选中选

Top-K 采样是一种简单粗暴的限制方法：**只保留概率最高的 K 个 Token，其余的全部设为 0（相当于排除）**，然后在剩下的 K 个中按概率采样。

假设 K = 3：

```
原始概率分布：
  "今天"  0.52  ✓ 保留
  "明天"  0.17  ✓ 保留
  "昨天"  0.09  ✓ 保留
  "天气"  0.02  ✗ 排除
  "苹果"  0.002 ✗ 排除
  ...（其余 10 万个 Token 全部排除）

重新归一化后：
  "今天"  0.67
  "明天"  0.22
  "昨天"  0.11
```

K 值的选择直接影响生成质量：

| K 值 | 效果 | 适用场景 |
|------|------|----------|
| K = 1 | 等价于贪心解码 | 需要确定性输出 |
| K = 5-10 | 输出较确定，略有变化 | 事实问答、代码生成 |
| K = 40-50 | 较好的平衡 | 一般对话 |
| K = 100+ | 高多样性 | 创意写作 |

**Top-K 的局限**：K 是固定的，不随语境变化。当模型非常确定时（某个词概率 0.99），你仍然保留了 K-1 个不该出现的词；当模型很犹豫时（前 5 个词概率都差不多），K=3 可能过早排除了好选项。

### Top-P（核采样）：自适应的候选集

Top-P 采样，也叫**核采样（Nucleus Sampling）**，解决了 Top-K 的固定候选问题。它的思路是：**按概率从高到低累加，当累计概率达到 P 时，就只从这些词中采样。**

假设 Top-P = 0.9：

```
概率从高到低排列：
  "今天"  0.52 → 累计 0.52 ✓
  "明天"  0.17 → 累计 0.69 ✓
  "昨天"  0.09 → 累计 0.78 ✓
  "天气"  0.08 → 累计 0.86 ✓
  "不错"  0.05 → 累计 0.91 ✓ 超过 0.9，截止！
  "苹果"  0.002 → 被排除
```

Top-P 的巧妙之处在于它是**自适应的**：

- 当模型很确定时（一个词概率 0.95），Top-P = 0.9 可能只保留 1-2 个词——输出非常确定。
- 当模型很犹豫时（十个词各约 0.1），Top-P = 0.9 会保留 9-10 个词——给了模型更多选择空间。

想象一张这样的图：概率分布的柱状图从左到右按高度递减排列。在 Top-K 模式下，一条竖线在第 K 根柱子处切下去，左边保留、右边砍掉，不管柱子高矮。在 Top-P 模式下，一条横线在累计概率等于 P 的位置画下来，柱子高于这条线的全部保留——高柱子多的时候保留少，矮柱子多的时候保留多。

> [!TIP]
> 实际使用中，**Top-P 是比 Top-K 更推荐的参数**。大多数现代 LLM API 的默认设置是 Temperature = 0.7、Top-P = 0.9。Top-K 和 Top-P 可以组合使用，但通常只开一个就够了。

### Beam Search：多路径搜索最优解

前面讨论的解码方法都是"走一步看一步"——每一步只做一个决定。Beam Search 是一种更全局的策略：**同时维护多条候选路径，最终选择整体得分最高的那条。**

想象你站在迷宫入口：

- **贪心解码**：每个岔路口都选看起来最对的那条路——走到哪儿算哪儿。
- **Beam Search**：每个岔路口保留 Beam Width 条最有希望的路径，一路走一路评估，最后选总分最高的。

以 Beam Width = 3 为例：

```
第 1 步：选出概率最高的 3 个起始词
  "今天"(0.52)  "明天"(0.17)  "昨天"(0.09)

第 2 步：每个词后面各扩展所有可能，产生 3×词表 个候选序列
  评估所有候选，保留累计概率最高的 3 条：
  "今天天气"(0.31)  "明天天气"(0.12)  "今天阳光"(0.10)

第 3 步：继续扩展，继续裁剪...
  "今天天气很好"(0.22)  "明天天气也不错"(0.08)  "今天阳光明媚"(0.07)

...直到所有路径生成结束符，选总分最高的那条。
```

Beam Search 的优缺点：

| 方面 | 表现 |
|------|------|
| 优点 | 考虑了全局最优，不会因为一步选错导致后面的句子崩坏 |
| 缺点 | 计算量是贪心解码的 Beam Width 倍；可能生成过于"安全"的平庸文本 |
| 适用场景 | 机器翻译、文本摘要等需要准确性的任务 |

> [!NOTE]
> 在开放式对话中，Beam Search 并不是最佳选择。因为它倾向于选择"平均而言最安全"的文本，结果往往比采样方法更无聊。但在翻译等需要精确性的任务中，Beam Search 仍然是主流方案。

### 推理加速：让模型跑得更快

大模型的推理有一个根本矛盾：**模型越大，能力越强，但生成越慢**。一个 70B 参数的模型，每生成一个 Token 都要遍历 700 亿个参数。如果你问它写一篇 2000 字的文章，可能要等上好几分钟。

这种速度对实时交互来说不可接受。于是，一系列推理加速技术被开发出来。

#### KV Cache：不要重复计算

这是 Transformer 推理中最重要的优化，没有之一。

理解 KV Cache 需要回忆 §05 中的自注意力机制。在计算当前词的注意力时，需要用到之前所有词的 Key 和 Value 向量。问题是，每生成一个新 Token 时，之前的 Token 都要重新算一遍 Q/K/V 吗？

不需要。之前 Token 的 K 和 V 已经算过了，**存下来就行**。

想象一张这样的图：一个不断增长的时间轴，左边是已经生成的 Token 序列，右边是正在生成的最新 Token。每一步，上方有一个"缓存区"，里面存储着之前所有 Token 的 K 和 V 向量。新生成的 Token 只需要计算自己的 Q/K/V，然后从缓存区读取之前的 K/V，就能完成注意力计算。同时，新 Token 的 K/V 也被追加到缓存区中。

```
生成第 1 个 Token：计算 K₁, V₁，存入缓存
生成第 2 个 Token：计算 K₂, V₂，从缓存读取 K₁, V₁，存入 K₂, V₂
生成第 3 个 Token：计算 K₃, V₃，从缓存读取 K₁, V₁, K₂, V₂，存入 K₃, V₃
...
```

KV Cache 的效果：对于长文本生成，推理速度可以提升 **几倍到几十倍**。没有 KV Cache，GPT-4 这样的模型在生成长文本时几乎不可用。

**KV Cache 的代价**：显存占用。每个 Token 的 K 和 V 都要存下来，如果上下文长度是 128K，那缓存区的显存占用可能超过模型本身。这也是为什么 GPU 显存对于推理如此重要。

> [!WARNING]
> KV Cache 的大小与上下文长度成正比。当上下文窗口越来越大（GPT-4 支持 128K，Claude 支持 200K），KV Cache 的显存压力成为推理系统的最大瓶颈之一。为此，业界开发了 MQA（Multi-Query Attention）和 GQA（Grouped-Query Attention）等技术，通过让多个注意力头共享同一组 K/V 来减少缓存大小。

#### 推测解码（Speculative Decoding）：用小模型猜、大模型验

KV Cache 解决了"不重复计算"的问题，但每次前向传播仍然要跑完整个大模型。推测解码的思路更激进：**用一个小模型先"猜"几个 Token，然后让大模型一次性验证。**

流程如下：

```
1. 小模型（比如 7B）快速生成 5 个 Token：A B C D E
2. 大模型（比如 70B）一次前向传播，同时验证这 5 个 Token
3. 假设前 3 个（A B C）通过了验证，后 2 个（D E）被否决
4. 保留 A B C，从大模型的概率分布重新采样 D
5. 重复上述过程
```

关键洞察：大模型一次前向传播可以同时得出所有位置的概率分布，验证多个 Token 并不比验证一个慢多少。如果小模型的猜测准确率高（比如 70%），那整体速度就能提升 2-3 倍。

想象一张这样的图：左边是一个"快但不太准"的小模型在快速写草稿，右边是一个"慢但精准"的大模型在审阅。小模型一口气写完一段话，大模型扫一眼，说"前三句没问题，后面不对，从这里重写"。两人配合，效率比大模型一个人从头写快得多。

#### 量化（Quantization）：用更小的数字

模型的权重通常是 16 位浮点数（FP16 或 BF16），每个参数占 2 字节。一个 70B 模型光存储权重就需要 140GB 显存。

量化的思路很简单：**用更少的位数来表示这些数字。**

| 精度 | 每参数占内存 | 70B 模型总大小 | 效果损失 |
|------|-------------|---------------|----------|
| FP16 | 2 字节 | 140 GB | 基准 |
| INT8 | 1 字节 | 70 GB | 极小 |
| INT4 | 0.5 字节 | 35 GB | 小幅 |
| INT3 | 0.375 字节 | ~26 GB | 可感知 |

一个类比：把高清照片压缩成 JPEG。肉眼几乎看不出区别，但文件大小缩小了好几倍。量化也是如此——把 16 位的精确数字压缩成 4 位，模型的绝大部分能力都保留了，但内存和计算需求大幅降低。

> [!TIP]
> 量化是目前**在消费级硬件上运行大模型的关键技术**。一个经过 INT4 量化的 70B 模型可以塞进一张消费级显卡（如 RTX 4090，24GB 显存），而原始精度需要多张专业显卡。GGUF 格式（被 llama.cpp 使用）就是为此设计的——它在 CPU 和 GPU 之间灵活分配计算，让你在普通笔记本上也能跑大模型。

### 批处理推理：同时服务多人

当你一个人在使用 ChatGPT 时，GPU 的大部分计算能力其实在"摸鱼"——因为生成一个 Token 只需要几毫秒的计算，但大量时间花在数据搬运上。

批处理推理的思路：**把多个用户的请求打包成一批，一起送进 GPU 计算。**

想象一张这样的图：一条传送带上有多个盒子（用户请求），它们被排列在一起送入一台大型机器（GPU）同时处理。如果是逐个处理，机器大部分时间在等；打包处理后，机器的利用率从 10% 跳到了 90%。

但批处理有一个挑战：不同用户的输入长度不同，生成速度也不同。有人问一句话就能回答，有人要模型写一篇长文。怎么办？

**连续批处理（Continuous Batching）** 解决了这个问题：

- 不等所有请求都完成才返回结果
- 某个请求完成后，立即把新请求填入空出的"槽位"
- GPU 始终保持满载工作

这是 vLLM、TGI 等现代推理框架的核心技术。它让单个 GPU 能同时服务几十个甚至上百个用户，大幅降低每个请求的边际成本。

> [!NOTE]
> 推理优化是一个活跃的研究领域。除了上述技术，还有 **Flash Attention**（优化注意力计算的显存访问模式）、**PagedAttention**（像操作系统管理虚拟内存一样管理 KV Cache）等创新。这些技术的组合使得大模型的推理成本在快速下降——2023 年到 2025 年，每百万 Token 的推理价格下降了超过 90%。

### 本章小结

让我们把推理与解码的全流程串起来：

```
用户输入 → 分词 → 嵌入 → 多层 Transformer → Logits 向量
                                                    ↓
                                            Softmax → 概率分布
                                                    ↓
                                    解码策略（Temperature/Top-K/Top-P）
                                                    ↓
                                            选出下一个 Token
                                                    ↓
                                    拼接到已有文本，重复上述过程
                                                    ↓
                                            直到生成结束符
```

核心要点：

1. **Softmax** 把原始分数转化为概率分布
2. **Temperature** 控制概率分布的"尖锐程度"，决定输出是保守还是大胆
3. **Top-K 和 Top-P** 限制采样范围，避免选中不合理的 Token
4. **Beam Search** 通过维护多条候选路径来寻找全局更优的输出
5. **KV Cache** 是推理加速的基石，避免了重复计算
6. **推测解码** 用小模型猜大模型验，巧妙提升速度
7. **量化** 通过降低数值精度来减少内存和计算需求

理解了解码过程，你就理解了大模型推理的完整生命周期。下一章，我们把视角拉到产品层面——训练好的模型怎样变成真正有用的产品。

## §12 从模型到产品——RAG、Agent 与未来
From Model to Product: RAG, Agents, and Beyond

### 从"能聊天"到"能干活"的跨越

经过前面十一章的旅程，我们已经理解了大模型的完整训练流程：预训练让它获得知识，微调让它学会对话，对齐让它变得安全。但训练好的模型本身只是一个"大脑"——它坐在服务器里，等着人提问，然后给出回答。

一个能回答问题的"大脑"已经很有价值了，但人们想要更多。我们希望它：

- 能**查资料**再回答，而不是凭记忆瞎说
- 能**动手操作**，而不是只会用文字指挥
- 能**看图、听声音**，而不是只认识文字
- 能**可靠、安全**地工作，而不是偶尔胡言乱语

这就是从"模型"到"产品"的跨越。本章将拆解这个跨越中最关键的技术和思想。

### Prompt Engineering：提问的艺术

在讨论任何复杂应用之前，我们先来看最基础也最实用的技能：**怎样向模型提一个好问题。**

Prompt Engineering 不是什么高深的技术，它更像是和一个非常聪明但需要明确指令的同事沟通的技巧。掌握几个核心原则，就能显著提升输出质量。

#### 原则一：明确角色和语境

模糊的 Prompt：

```
帮我写一篇关于量子计算的文章。
```

明确的 Prompt：

```
你是一位面向大众的科技科普作者。请写一篇 800 字左右的量子计算入门文章。
目标读者是没有物理背景的大学生。请用生活中的类比来解释核心概念，
避免使用数学公式。文章结构：引言 → 什么是量子比特 → 量子计算的优势 →
目前的挑战 → 展望。
```

区别在于：后者告诉了模型**你是谁、写给谁看、写什么、怎么写**。这就像给一个自由撰稿人下达任务——背景越清楚，产出越接近你的预期。

#### 原则二：给示范（Few-Shot）

有时候，说一百句不如给一个例子。Few-Shot Prompting 就是在 Prompt 中给出几个输入-输出的示例，让模型"照着样子做"。

```
请根据以下格式，将用户的自然语言转换为 SQL 查询：

示例 1：
用户：查找所有年龄大于 30 的员工
SQL：SELECT * FROM employees WHERE age > 30;

示例 2：
用户：统计每个部门的平均工资
SQL：SELECT department, AVG(salary) FROM employees GROUP BY department;

现在请处理：
用户：找出工资最高的前 5 名员工的名字和部门
SQL：
```

模型会按照示例的模式生成正确的 SQL。这在格式化输出、分类任务、格式转换等场景中非常有效。

#### 原则三：让模型"想清楚再回答"（Chain-of-Thought）

一个经典的发现：当你让模型"一步一步地思考"时，它在推理任务上的表现会大幅提升。

```
请一步一步地分析以下问题，展示你的推理过程：
一个游泳池有两个进水管和一个出水管。A 管单独注满需要 4 小时，
B 管单独注满需要 6 小时，出水管单独放完需要 12 小时。
如果三管同时打开，多久能注满？
```

Chain-of-Thought（思维链）的威力在于：它让模型的中间推理过程变得可见。这不仅提高了答案的正确率（因为推理错误可以在过程中被发现和修正），也让结果更可信（用户可以审查推理步骤）。

> [!TIP]
> Prompt Engineering 的核心心法：把模型当成一个能力极强但完全不了解你背景的合作伙伴。你需要告诉它一切它需要知道的信息——你是谁、你要什么、什么格式、什么风格、什么约束。信息越充分，输出越精准。

### RAG（检索增强生成）：给模型一个外挂知识库

#### 为什么需要 RAG

大模型有两个致命的知识缺陷：

**第一，知识有截止日期。** GPT-4 的训练数据截止到 2023 年，它不知道 2024 年发生了什么。你问它最新的 iPhone 评测，它会诚实地告诉你"我不知道"。

**第二，它不知道你公司的内部信息。** 你们的内部文档、客户资料、产品规范——这些全都没出现在训练数据中。模型再聪明，也回答不了"上周五发的那个版本有什么已知 Bug"。

**第三，它会编造。** 当模型不知道答案时，它不是沉默，而是"编"一个看起来像那么回事的答案。这就是大名鼎鼎的**幻觉（Hallucination）**问题。


![图12-1：RAG（检索增强生成）架构流程](images/rag-architecture.png)
RAG（Retrieval-Augmented Generation，检索增强生成）是解决这些问题的主流方案。

#### RAG 的完整流程

想象一张这样的图：图的左侧是一个"知识库"（一堆文档），中间是一个"检索器"（搜索引擎），右侧是"大模型"（生成器）。用户的提问先经过检索器，从知识库中找到相关段落，然后把这些段落和用户的问题一起交给大模型，让大模型基于这些材料来回答。

具体分为五个步骤：

**步骤一：知识准备——建库**

把你的文档（PDF、Word、网页、数据库记录等）切成小段（通常每段 200-500 个 Token），然后把每一段转换成一个向量（通过嵌入模型）。这些向量被存入**向量数据库**——一种专门存储和检索向量的数据库。

```
原始文档："我们的退货政策是购买后 30 天内可以无条件退货。退款将在 5-7 个工作日内到账。"
    ↓ 切片
段落 1："我们的退货政策是购买后 30 天内可以无条件退货。"
段落 2："退款将在 5-7 个工作日内到账。"
    ↓ 嵌入
段落 1 → [0.12, -0.34, 0.56, ..., 0.78]  （1536 维向量）
段落 2 → [-0.23, 0.45, -0.12, ..., 0.34]  （1536 维向量）
    ↓ 存入向量数据库
```

**步骤二：用户提问——检索**

当用户问"退货要多久能收到退款？"时：

1. 把这个问题也转换成向量：`[0.15, -0.30, 0.52, ..., 0.76]`
2. 在向量数据库中搜索与这个向量最相似的段落（通过余弦相似度等度量）
3. 返回最相关的 Top-K 个段落（通常是 3-10 个）

```
用户问题："退货要多久能收到退款？"
    ↓ 嵌入
问题向量：[0.15, -0.30, 0.52, ..., 0.76]
    ↓ 向量搜索
最相似段落：
  1. "退款将在 5-7 个工作日内到账。"（相似度 0.92）
  2. "我们的退货政策是购买后 30 天内可以无条件退货。"（相似度 0.71）
```

**步骤三：拼接上下文——增强**

把检索到的段落和用户问题拼接在一起，形成增强后的 Prompt：

```
请根据以下参考资料回答用户的问题。如果资料中没有相关信息，
请明确说"根据已有信息无法回答"，不要编造。

参考资料：
1. "退款将在 5-7 个工作日内到账。"
2. "我们的退货政策是购买后 30 天内可以无条件退货。"

用户问题：退货要多久能收到退款？
```

**步骤四：大模型生成——回答**

模型基于提供的参考资料生成回答。因为有参考资料在手，模型的回答有根有据，幻觉风险大大降低。

**步骤五：持续更新——维护**

知识库可以随时更新——新增文档、删除过期信息、修改现有内容。不需要重新训练模型，只需要更新向量数据库即可。这是 RAG 相对于微调最大的优势：**知识的更新成本几乎为零。**

> [!NOTE]
> RAG vs 微调，不是二选一，而是互补。简单来说：**知识类需求用 RAG，行为类需求用微调。** 如果你需要模型回答最新的公司政策，用 RAG；如果你需要模型用特定的语气和格式回答，用微调。很多生产系统两者结合使用。

#### RAG 的进阶优化

基础的 RAG 能解决 60% 的问题，剩下的 40% 需要各种优化策略：

- **查询改写**：用户的原始问题可能表达不清，先用模型把问题改写得更精确，再去做检索。
- **重排序（Reranking）**：向量检索速度快但不够精确，可以先用向量检索召回大量候选，再用一个专门的排序模型精排。
- **混合检索**：结合向量搜索和关键词搜索，取长补短。
- **分块策略优化**：切文档的方式直接影响检索质量。按语义边界切比按固定长度切效果好得多。

### Agent：让模型学会"动手"

如果说 RAG 让模型长了"记忆"，Agent 就是让模型长了"手"和"脚"。

一个纯粹的语言模型只能输入文字、输出文字。它能告诉你怎么做披萨，但不能帮你点外卖。Agent 的核心思想是：**给模型配备工具，让它能执行实际操作。**

#### 工具调用（Tool Use）

想象一张这样的图：中间是"大模型"这个大脑，周围通过线条连接着各种"工具"——一个搜索图标代表网络搜索，一个计算器图标代表代码执行，一个数据库图标代表查询数据库，一个日历图标代表查看日程。大模型像指挥官一样，根据任务需要调用不同的工具。

一个典型的工具调用流程：

```
用户：帮我查一下明天北京的天气，如果气温低于 10 度，
      就发一封邮件给 team@example.com 提醒大家多穿衣服。

模型的思考过程：
1. 我需要查询天气 → 调用天气 API
   工具调用：get_weather(city="北京", date="明天")
   返回结果：气温 5°C，晴

2. 气温 5°C < 10°C → 需要发邮件
   工具调用：send_email(to="team@example.com",
               subject="明天降温提醒",
               body="大家好，明天北京气温仅 5°C，请注意保暖。")
   返回结果：发送成功

3. 向用户报告结果
   回复："已经查了天气，明天北京 5°C。邮件已经发送给团队了。"
```

模型在这里不只是回答问题——它在**规划步骤、调用工具、根据中间结果做决策**。这就是 Agent 的核心能力。

#### ReAct 框架：推理与行动的循环

最经典的 Agent 模式是 **ReAct（Reasoning + Acting）**。它让模型在"思考"和"行动"之间交替循环：

```
思考 1：用户想知道中国人口最多的城市。我需要搜索这个信息。
行动 1：search("中国人口最多的城市")
观察 1：上海是中国人口最多的城市，常住人口约 2487 万。
思考 2：已经得到了答案。让我整理一下回复用户。
回答：中国人口最多的城市是上海，常住人口约 2487 万。
```

每一轮循环包含三个步骤：
1. **思考（Thought）**：分析当前状态，决定下一步做什么
2. **行动（Action）**：调用一个工具执行操作
3. **观察（Observation）**：获取工具返回的结果

这个循环一直持续，直到模型认为已经收集了足够的信息来回答用户的问题。

#### 规划与反思

更高级的 Agent 不只是"一步步走"，而是能做**整体规划**和**自我反思**。

**规划（Planning）**：面对复杂任务，Agent 先把任务拆成子任务，制定执行计划，然后逐步执行。

```
用户：帮我准备一份关于 AI 在医疗领域应用的演示文稿。

Agent 的规划：
1. 调研 AI 在医疗领域的最新应用（搜索工具）
2. 整理关键案例和数据（笔记工具）
3. 生成演示文稿大纲（文本生成）
4. 根据大纲创建幻灯片（PPT 工具）
5. 生成配套演讲稿（文本生成）
```

**反思（Reflection）**：Agent 执行完一步后，检查结果是否合理，如果不好就调整策略重新来。

```
步骤 3 的初步大纲：
  1. AI 简介
  2. 医疗 AI 案例
  3. 未来展望

Agent 反思：这个大纲太笼统了。让我重新搜索一些具体案例，
           然后生成一个更有深度的版本。
```

> [!WARNING]
> Agent 目前还处于早期阶段。让大模型自主调用工具存在风险——它可能调用错误的工具、传入错误的参数、或者陷入无限循环。生产环境中需要严格的权限控制和人工审批机制。不要让 Agent 有你没有审查过的"_delete"权限。

### 多模态：从文字到图像、音频、视频

到目前为止，我们讨论的模型都在处理文字。但人类感知世界的方式远不止文字——我们看图、听声音、看视频。**多模态大模型**正在让 AI 获得同样的能力。

多模态的核心挑战是：**不同模态的信息怎么"对齐"？** 图像是像素网格，音频是波形序列，文本是离散 Token——它们的"语言"完全不同。

解决思路是一个统一的框架：

```
图像 → 视觉编码器 → 图像 Token 序列    ─┐
                                          ├→ 统一 Token 流 → Transformer → 输出
文本 → 文本 Tokenizer → 文本 Token 序列 ─┘
```

把不同模态的信息都转化成 Token，然后喂给同一个 Transformer 模型。模型在统一的 Token 空间中学习不同模态之间的对应关系。

想象一张这样的图：三条输入管道——文字管道、图像管道、声音管道——从不同方向汇入同一个"大熔炉"（Transformer 模型）。每个管道把各自的输入转化为统一格式的小方块（Token），这些方块混在一起进入熔炉，出来的是一个融合了多种信息的输出。

多模态的实际能力：

- **图像理解**：GPT-4V、Claude 3 可以看图回答问题——"这张图表的趋势是什么？""这个代码截图有什么 Bug？"
- **图像生成**：DALL-E 3、Midjourney 根据文字描述生成图片
- **语音理解与生成**：Whisper 做语音转文字，GPT-4o 做实时语音对话
- **视频理解**：Gemini 1.5 可以分析长达 1 小时的视频内容

> [!TIP]
> 多模态不只是"加了几个功能"这么简单。它改变了人机交互的基本方式——你可以直接给模型看截图问"这个界面怎么改更好"，也可以拍照一道数学题让模型讲解。这种交互方式比纯文字自然得多，是多模态模型的真正价值所在。

### 大模型的局限：幻觉、偏见、安全

在欢呼大模型能力的同时，我们必须清醒地认识它的局限。理解局限不是为了否定它，而是为了更合理地使用它。

#### 幻觉（Hallucination）

大模型最广为人知的问题是**幻觉**——一本正经地胡说八道。模型会编造不存在的事实、虚构不存在的论文引用、捏造不存在的 API 方法。

幻觉的根源在于模型的本质：**它是概率预测器，不是知识检索器。** 当模型不确定答案时，它不是返回"我不知道"，而是生成"看起来最可能"的文本。在概率的驱动下，"编一个听起来合理的答案"比"承认不知道"的"得分"更高。

缓解幻觉的策略：

| 策略 | 原理 | 效果 |
|------|------|------|
| RAG | 提供外部知识，让模型基于事实生成 | 显著减少 |
| 要求模型引用来源 | 迫使模型在回答时引用证据 | 中等 |
| 多次采样比对 | 生成多个回答，检查一致性 | 辅助判断 |
| 明确说"不知道也没关系" | 在 Prompt 中明确允许模型说"我不确定" | 有一定帮助 |

> [!WARNING]
> 幻觉无法被完全消除。这是大语言模型架构的固有特性——只要它是基于概率生成文本，就有可能生成不真实的内容。在高风险场景（医疗、法律、金融）中使用大模型时，必须有人类专家的审核环节。

#### 偏见（Bias）

大模型从训练数据中学习，而训练数据来自互联网。互联网上的内容充满了偏见——性别偏见、种族偏见、文化偏见。模型不可避免地会学到这些偏见。

例如，如果训练数据中"程序员"这个词大多与男性关联，模型可能在生成关于程序员的文本时默认使用"他"而不是"她"。

缓解偏见需要多管齐下：训练数据的清洗和平衡、对齐训练中的偏见纠正、推理时的输出过滤。但这不是一个可以完全解决的技术问题——它需要持续的社会对话和技术改进。

#### 安全风险

大模型的安全风险是多维度的：

- **恶意使用**：生成钓鱼邮件、虚假信息、恶意代码
- **隐私泄露**：模型可能从训练数据中"记住"个人信息，在生成时泄露
- **越狱攻击**：通过巧妙的 Prompt 绕过安全限制，让模型执行不该执行的操作
- **过度依赖**：用户盲目信任模型输出，不做验证

这些风险需要技术手段（安全对齐、输出过滤、使用监控）和制度手段（使用规范、法律法规、行业自律）共同应对。

### 未来展望

站在 2025-2026 年的时间节点，大模型技术仍在高速演进。几个值得关注的趋势：

**趋势一：效率革命。** 模型不一定要更大才能更强。通过更好的架构设计（如混合专家模型 MoE）、更高效的训练方法、更聪明的推理优化，中小模型正在追赶大模型的能力。一个 2026 年的 7B 模型可能已经达到 2023 年 70B 模型的水平。

**趋势二：Agent 化。** 大模型正在从"回答问题的工具"进化为"能执行复杂任务的助手"。代码编写、数据分析、网页操作——越来越多的工作可以由 Agent 完成。但安全性和可控性仍然是核心挑战。

**趋势三：多模态融合。** 文字、图像、音频、视频的边界正在模糊。未来的模型将能同时理解和生成多种模态的内容，让交互更加自然和丰富。

**趋势四：垂直领域深化。** 通用模型的天花板逐渐显现，而经过领域数据深度训练的专业模型（医疗、法律、金融、编程）正在展现出更大的实用价值。

**趋势五：开源生态繁荣。** LLaMA、Mistral、Qwen 等开源模型的能力持续提升，社区驱动的创新速度越来越快。开源与闭源的竞争正在推动整个行业向前发展。

### 全书总结：从一个猜词游戏到改变世界的技术

让我们回到 §01 的起点。

大语言模型的全部秘密，浓缩成一句话就是：**通过在海量文本上练习"猜下一个词"，模型学会了语言的模式，而这些模式中编码了关于世界的知识、逻辑和推理能力。**

回顾我们的旅程：

**Part 1（基础构件）** 我们拆解了模型的输入——Token、嵌入、位置编码，以及 Transformer 这个革命性的架构。你理解了模型"看到"的文字世界和人类看到的有多么不同。

**Part 2（预训练）** 我们深入了模型的学习过程——自注意力机制如何捕捉全局关联，预训练如何从海量数据中提取知识，以及 Scaling Law 为什么"更大就是更好"。你理解了为什么"大力出奇迹"不只是一句口号，而是有数学基础的科学规律。

**Part 3（微调与对齐）** 我们看到了模型从"博学但不听话"到"既聪明又安全"的转变。SFT 教会了它对话，RLHF 和 DPO 教会了它什么回答是人类偏好的。你理解了为什么对齐是不可或缺的环节。

**Part 4（推理与应用）** 我们走完了从模型到产品的最后一公里——解码策略如何影响输出质量，RAG 如何解决知识时效性问题，Agent 如何让模型从"说"变成"做"。你理解了把模型变成真正有用的产品需要什么。

大模型不是魔法。它是数学、工程和数据的精密结合。理解了原理，你就不会再被"AI 将取代一切"或"AI 不过如此"这两种极端声音所迷惑。你会知道它能做什么、不能做什么、以及怎样更好地利用它。

这个世界不需要每个人都去训练大模型，但需要更多人理解它的原理。因为只有在理解的基础上，我们才能做出明智的判断——什么时候该信任它，什么时候该质疑它，什么时候该用它，什么时候该超越它。
