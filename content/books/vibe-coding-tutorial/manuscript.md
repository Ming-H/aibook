# Part 1: 概念
## §01 从写代码到说代码：Vibe Coding 是什么
From Writing Code to Speaking Code: What Is Vibe Coding
2025 年 2 月，Andrej Karpathy 发了一条推文：
> There's a new kind of coding I call "vibe coding", where you fully give in to the vibes, embrace exponentials, and forget that the code even exists.
这条推文炸了。一个月内，"Vibe Coding"成了科技圈最热的词。到了年底，Collins 词典把它评为 2025 年度词汇。
你的第一反应可能是：又一个 buzzword？AI 编程助手我早就在用了，Copilot 补全代码都补了两年了，有什么新鲜的？
还真有。
### 不是 Copilot 换了个名字
Vibe Coding 和传统的 AI 代码补全，区别不在工具，在心态。
Copilot 时代：你写代码，AI 帮你补全下一行。你是司机，AI 是副驾驶，帮你踩踩油门。
Vibe Coding 时代：你用自然语言描述想要什么，AI 写完整段代码。你是乘客，AI 是导航，你只需要说"我要去哪"。
打个比方。你用过 GPS 导航吧？你不需要记住每条路怎么走，只需要告诉导航目的地。但——你不能闭着眼开。导航说右转，你也得看一眼是不是右转进了河里。
这就是 Vibe Coding 的核心张力：**信任 AI，但别盲信。**
| 维度 | 传统 AI 补全 | Vibe Coding |
|------|-------------|-------------|
| 你做什么 | 写代码主体，AI 补全细节 | 描述需求，AI 生成代码 |
| AI 的角色 | 副驾驶 | 导航系统 |
| 交互方式 | Tab 接受建议 | 自然语言对话 |
| 代码归属 | 你写的，AI 辅助 | AI 写的，你审查 |
| 心态 | 我在写代码 | 我在和 AI 协作 |
| 典型工具 | GitHub Copilot | Cursor、Claude Code、Windsurf |
### 三个真实场景
光说概念没感觉，看三个场景：
**场景一：产品经理要做原型。** 不用找开发排期了，打开 Cursor，说"做一个登录页面，支持邮箱和手机号"，10 分钟出原型。能跑。不能上生产，但够用来做用户测试。
**场景二：后端开发者写脚本。** 每次部署要手动改配置文件、压缩资源、上传服务器。跟 Claude Code 说"帮我写个部署脚本，自动完成这些步骤"，5 分钟拿到一个能用的 shell 脚本。
**场景三：独立开发者做副业。** 有个想法但不想从零写代码。用 Bolt.new 描述需求，30 分钟出一个 MVP（Minimum Viable Product，最小可行产品）。不完美，但足以验证想法值不值得继续投入。
这三个场景的共同点：**人关注"做什么"，AI 处理"怎么做"。**
### GPS 隐喻：贯穿全书的理解框架
我选了一个隐喻来帮你理解 Vibe Coding：**带着 GPS 开车。**
| 开车场景 | Vibe Coding 概念 |
|----------|-----------------|
| 目的地 | 项目需求 / 你想要的产品 |
| GPS 导航 | AI 编程助手 |
| 驾驶技术 | 你的编程基础 |
| 导航规划的路线 | AI 生成的代码方案 |
| 走错路重新规划 | 迭代、调试、修改 Prompt |
| 路况判断 | 代码审查、安全检查 |
| 盲目跟导航开进河里 | 不看代码直接用 AI 输出 |
这个隐喻会贯穿全书。当你读到后面讨论工具选型、质量把控、迭代策略的时候，想想"如果是开车，这时候我该做什么"。
> **核心建议**
>
> Vibe Coding 不是"不会写代码的人也能写代码了"。它是"会写代码的人可以写得更快、做得更多了"。GPS 不能替你考驾照，但有了驾照之后，GPS 让你敢去没去过的地方。
下一节拆开讲：Vibe Coding 和你已经在用的 AI 工具到底差在哪。
## §02 不只是自动补全：Vibe Coding 的真正边界
Beyond Autocomplete: The Real Boundaries of Vibe Coding
你可能觉得：我用 Copilot 补全代码补了两年，这不就是 Vibe Coding 吗？
不是。差别比你想的大。
### 从"一行建议"到"一段实现"
Copilot 的工作模式：你打了一行 `def fetch_user(`，它猜你想写什么，给出补全建议。你按 Tab，接受。它帮你省了几次敲键盘。
Vibe Coding 的工作模式：你说"写一个函数，从数据库获取用户信息，支持分页和模糊搜索，返回 JSON 格式"，AI 给你完整实现。包含错误处理、参数校验、返回格式。你可能需要调整，但起点完全不同。
这不是量的差异，是质的跳跃。
| 维度 | Copilot 模式 | Vibe Coding 模式 |
|------|-------------|-----------------|
| 输入 | 代码上下文 | 自然语言需求 |
| 输出 | 下一行/几行建议 | 完整函数/模块 |
| 交互频率 | 每几秒一次 | 每几分钟一次 |
| 人的角色 | 写代码 + 审核 AI 补全 | 描述需求 + 审核 AI 生成 |
| 上下文窗口 | 当前文件 | 整个项目 |
| 思维方式 | "我要写什么代码" | "我要实现什么功能" |
### 四个层级：你在哪一层
AI 辅助编程不是一刀切。它有四个层级，层层递进：
### 补全与对话
**Level 1：行内补全。** 就是 Copilot 的经典模式。你写一半，它补另一半。省打字时间。
**Level 2：对话式辅助。** 你在编辑器里打开一个对话窗口，问"这个报错怎么修"，AI 看你的代码上下文给建议。不用离开编辑器去搜 Stack Overflow。
### Agent 与全自动
**Level 3：Agent 模式。** 你给一个任务描述，AI 自己规划步骤、读写文件、运行命令、查看结果。你只需要在关键节点确认。Claude Code、Cursor Agent 都在做这个。
**Level 4：全自动。** 你说"做一个 XX"，AI 从设计到部署全搞定。目前还没完全到这一步，但 Bolt.new、Replit Agent 已经在逼近。
你现在大概率在 Level 1 或 2。本教程的目标是帮你稳稳站在 Level 3，并知道 Level 4 的边界在哪。
![四个层级的 AI 编程](images/four-levels-ai-programming.png)
### 它能做什么，不能做什么
先泼冷水。Vibe Coding 不是万能的。
**擅长的：**
- 快速原型和 MVP
- 重复性的 CRUD（增删改查，Create/Read/Update/Delete）逻辑
- 脚本和自动化任务
- 已有代码的重构和优化
- 学习新技术时的实验项目
**不擅长的：**
- 高并发系统的核心架构设计
- 需要深度领域知识的业务逻辑
- 对安全性有极端要求的模块
- 需要精细性能调优的代码
- 你自己都说不清楚的需求
最后一条最关键。GPS 再好，你说不清楚目的地，它也没法导航。Vibe Coding 的天花板不是 AI 的能力，是你描述需求的能力。
> **核心建议**
>
> 如果你只是想 Tab 补全快点，Copilot 够了。如果你想让 AI 帮你实现完整功能，你需要学会 Vibe Coding 的核心技能：把想法说清楚、审查 AI 的输出、在出错时高效迭代。这三样就是本教程要教你的。
下一节，我们聊聊上路之前你需要准备什么。
## §03 上路之前：你需要什么基础和心态
Before You Hit the Road: Prerequisites and Mindset
"我不会写代码，能用 Vibe Coding 吗？"
能。但你的体验会和会写代码的人完全不同。
这就像问"我不会开车，能用导航吗？"——当然能，但你得先学会开车。导航不能替你踩刹车。
### 你需要的技术底子
说实话，不需要很多，但需要有一些。
| 你需要会的 | 为什么需要 | 最低要求 |
|-----------|-----------|---------|
| 看懂代码 | AI 写的代码你要能判断对不对 | 能读懂 Python/JS 的基本逻辑 |
| 用命令行 | Vibe Coding 工具大量在终端操作 | 会 cd、ls、python node 这些基本命令 |
| 理解项目结构 | AI 会在你的项目里改文件 | 知道什么是目录、配置文件、入口文件 |
| 基础调试能力 | AI 写的代码不会一次就对 | 能看报错信息，大概知道问题在哪 |
| Git 基础 | 你需要频繁提交和回滚 | 会 add、commit、push、reset |
如果你的编程经验在 1 年以上，这些应该都没问题。如果完全没有编程经验，建议先花一两周学个 Python 入门，再来看这本教程。
### 心态调整：三要三不要
Vibe Coding 需要的心态和传统编程很不一样。调不好心态，工具再好也用不出效果。
### 三要
**要：信任 AI 的第一版输出。**
不要一上来就逐行审查。先让它跑起来，看效果。有问题再改。GPS 给你路线，你先跟着走，发现不对再调整。
**要：快速迭代，不要完美主义。**
Vibe Coding 的核心优势是快。你提需求，AI 生成，你看效果，改需求，AI 重新生成。每个循环 1-5 分钟。别在一个版本上纠结太久。
**要：关注架构，放手细节。**
你来决定项目结构、技术选型、模块划分。让 AI 来写具体实现。你是规划路线的人，AI 是开车的。
### 三不要
**不要：不看代码就用。**
AI 生成的代码可能有 bug、安全漏洞、性能问题。你可以不写，但你必须能看懂。
**不要：一次提太大的需求。**
"帮我做一个电商网站"——太大了。"帮我做一个商品列表页面，从数据库读取数据，支持分页"——这差不多。把大需求拆成小任务，一个个来。
**不要：对 AI 说"随便"、"都行"。**
AI 需要明确的指令。"用 React 还是 Vue？""Python 还是 Node？"——你必须有主见。GPS 问你去哪，你说"随便"，它没法导航。
### 工具清单
接下来的章节会详细教你怎么用。这里先列个清单，你有印象就行：
| 工具 | 类型 | 用途 | 价格 |
|------|------|------|------|
| Cursor | AI IDE | 日常开发主力 | $20/月 |
| Claude Code | 终端 Agent | 复杂项目、深度重构 | 按量付费 |
| Windsurf | AI IDE | 轻量级替代 Cursor | $15/月 |
| GitHub Copilot | 编辑器插件 | 行内补全 | $10/月 |
| Bolt.new | Web 平台 | 快速原型 | $20/月 |
| Replit | Web 平台 | 在线编码+部署 | $25/月 |
不需要全装。本教程的示例主要用 Cursor 和 Claude Code，选一个主力工具跟着做就行。
> **核心建议**
>
> 心态比工具重要。会用 Cursor 的人很多，但能高效使用 Vibe Coding 的人很少。差别在于：你是把 AI 当打字机用，还是把它当协作搭档用。打字机你按一个键出一个字；搭档你交代一个任务，它给你一个方案，你们再一起打磨。

# Part 2: 核心机制
## §04 Prompt 即方向盘：怎么说 AI 才听得懂
Prompt as the Steering Wheel: How to Talk So AI Understands
"帮我写个网站。"
AI 真的会给你写个网站。但大概率不是你想要的那个。
Vibe Coding 里最核心的技能不是编程，是表达。你能不能把脑子里的想法说清楚，直接决定了 AI 输出的质量。
这不是玄学。有套路。
### 一个好 Prompt 的结构
经过大量实践，我发现一个好 Prompt 包含四个要素：
### 四要素拆解
**1. 角色（Role）** — 你希望 AI 扮演什么角色
"你是一个有 10 年经验的 Python 后端开发者" 比 "帮我写个 Python 函数" 效果好很多。
**2. 上下文（Context）** — 现在的状况是什么
"这是一个 Flask 项目，使用 SQLAlchemy ORM（Object-Relational Mapping，用对象操作数据库的中间层），数据库是 PostgreSQL" 比 "帮我写个数据库查询" 更精准。
**3. 任务（Task）** — 具体要做什么
"写一个用户注册接口，需要邮箱验证，密码用 bcrypt（一种密码加密算法）加密，返回 JWT（JSON Web Token，一种身份验证令牌）Token" 比 "写个注册功能" 更容易得到好结果。
**4. 约束（Constraints）** — 有什么限制条件
"不要用第三方库"、"代码要兼容 Python 3.8"、"注释用中文"——这些限制帮助 AI 缩小搜索空间。
### 合体示例
合在一起：

```text
你是一个有 10 年经验的 Python 后端开发者。
这是一个 Flask 项目，使用 SQLAlchemy ORM，数据库是 PostgreSQL。
写一个用户注册接口，需要邮箱验证，密码用 bcrypt 加密，返回 JWT token。
不要使用 Flask-RESTful 扩展，代码需要兼容 Python 3.10+。
```

### 五个常见错误
实际用下来，80% 的差结果来自这五个错误：
| 错误 | 示例 | 问题 | 怎么改 |
|------|------|------|--------|
| 太模糊 | "做个好看的前端" | AI 猜你要什么 | 指定：配色方案、布局方式、参考网站 |
| 太大 | "做个电商系统" | AI 一次吃不下 | 拆成：商品列表→购物车→订单→支付 |
| 没上下文 | "这个报错怎么修" | AI 不知道你的环境 | 附上：框架版本、文件路径、完整报错 |
| 相互矛盾 | "要快但也要安全" | AI 不知道优先什么 | 排优先级："安全优先，性能其次" |
| 一次要太多 | "顺便把测试也写了" | 多任务质量都下降 | 一次一个任务，完成后追加下一个 |
### 渐进式 Prompt：从模糊到精确
你可能不会一次写出完美的 Prompt。没关系。Vibe Coding 本身就是迭代的过程。
### 从粗到细
**第一步：粗描述。**
"帮我写一个文件上传功能。"
**第二步：加细节。**
"帮我写一个文件上传功能，支持图片和 PDF，最大 10MB，上传到本地 storage 目录。"
### 加约束和异常
**第三步：加约束。**
"帮我写一个文件上传功能，支持图片和 PDF，最大 10MB，上传到本地 storage 目录。用 FastAPI 实现，文件名用 UUID 重命名，返回文件的访问 URL。"
**第四步：加异常处理。**
在上面的基础上加："需要处理文件过大、格式不支持、存储空间不足的情况，返回合适的 HTTP 状态码。"
每一步 AI 的输出质量都在提升。你不需要一次到位，但你需要在看到结果后知道往哪个方向补。
### 一个实战对比
同样的需求，两种写法：
**差版本：**

```text
写个 TODO 应用
```

**好版本：**

```python
你是一个前端开发者，擅长 React + TypeScript。
写一个 TODO 应用，功能如下：
1. 添加待办事项（标题 + 优先级）
2. 标记完成
3. 按优先级排序
4. 本地存储，刷新不丢数据
技术要求：用 React hooks，不用 class 组件。
样式用 Tailwind CSS，配色偏蓝色系。
不需要后端，数据存 localStorage。
```

> **核心建议**
>
> Prompt 工程不是"提示词模板"，是"把需求想清楚"的能力。写好 Prompt 的过程，本质上是在整理你自己的思路。如果你说不清楚要什么，说明你还没想清楚。想清楚了，说给 AI 听自然就清楚。
下一节，我们打开引擎盖，看看 AI 编程工具到底是怎么工作的。
## §05 引擎盖下：AI 编程工具的工作原理
Under the Hood: How AI Coding Tools Work
你不需要理解 Transformer 架构才能用 Vibe Coding。就像你不需要理解发动机原理才能开车。
但你需要知道一些关键概念。因为理解了这些，你就知道什么时候该信任 AI，什么时候该自己上手。
### 三个关键组件
一个 AI 编程工具的内部，有三个关键部分在协作：
### 发动机：LLM
**1. 大语言模型（LLM）— 发动机。**
这是核心。Claude、GPT-4、Gemini 都是 LLM。它们通过阅读海量代码学会了"给定上下文，预测接下来应该写什么代码"。
关键特性：它不是在"搜索"代码，是在"生成"代码。每次输出都是根据你的 Prompt 和上下文即时生成的。这意味着同一个 Prompt 可能得到不同结果。
### 传感器：上下文
**2. 上下文系统 — 传感器。**
AI 怎么知道你的项目长什么样？靠上下文系统。不同的工具收集上下文的方式不同：
| 工具 | 上下文来源 |
|------|-----------|
| Copilot | 当前打开的文件 + 光标附近的代码 |
| Cursor | 当前文件 + 你手动引用的文件 + 项目索引 |
| Claude Code | 自动搜索相关文件 + 读取 CLAUDE.md（一个 Markdown 配置文件，记录项目规范和 AI 协作偏好）配置 |
| Bolt.new | 你描述的项目结构 + 已生成的代码 |
上下文越多，AI 的输出越准确。但上下文也有上限（叫"上下文窗口"），塞太多反而会分散注意力。
### 手和脚：工具调用
**3. 工具调用系统 — 手和脚。**
光能生成代码不够，AI 还得能把代码写进文件、运行命令、查看结果。这就是工具调用。
Cursor 的 Agent 模式可以：读取文件、创建文件、运行终端命令、搜索代码。
Claude Code 可以：读写文件、运行 bash 命令、搜索代码库、调用外部工具（通过 MCP（Model Context Protocol，一种让 AI 调用外部工具的开放协议））。
这些能力让 AI 从"只能建议"变成了"能执行"。
![AI 编程工具三大组件](images/ai-coding-components.png)
### 为什么 AI 会犯错
AI 编程工具犯错的根本原因有三个：
### 上下文不够
**1. 上下文不够。**
它没看到你项目里的某个关键文件，所以不知道你已经在 utils.py 里写了一个类似的函数，又造了一个轮子。
### 知识截止
**2. 知识截止。**
训练数据有时间限制。如果你用的是 2024 年训练的模型，它可能不知道某个库在 2025 年做了重大 API 变更。
### 统计偏差
**3. 统计特性。**
LLM 本质上是在预测"下一个 Token 最可能是什么"。有时候"最可能"不等于"正确"。特别是处理边界情况和特殊逻辑的时候。
理解了这三个原因，你就知道怎么应对了：
- 上下文不够 → 主动提供相关文件和信息
- 知识过时 → 告诉 AI 你用的版本号
- 统计偏差 → 对关键逻辑手动验证
### 对话上下文管理
和 AI 的对话会越来越长。AI 在对话中看到的每一条消息都会影响它的输出。这意味着：
**前几轮对话的质量，决定了后面的质量。**
如果你一开始给了清晰的项目结构和需求描述，后面的每轮对话 AI 都能保持一致的上下文。如果你一开始就模糊，后面会越来越偏。
> **核心建议**
>
> 把 AI 当成一个能力很强但对你项目一无所知的新同事。你不会让新同事直接改代码，你会先给他介绍项目结构、技术栈、代码规范。对 AI 也一样。项目介绍越清楚，协作越高效。
下一节讲最重要的主题：怎么审查 AI 生成的代码，确保质量。
## §06 别盲信导航：代码审查与质量把控
Don't Blindly Trust the GPS: Code Review and Quality Control
2025 年的一项研究数据：AI 生成的代码中，约 45% 包含安全漏洞。
这个数字可能让你紧张。应该紧张。但紧张不等于不用，而是要建立一套审查流程。
回想 GPS 隐喻：导航说右转，你扫一眼路况再转。不是为了不相信导航，是为了安全。
### AI 生成代码的五大类问题
| 问题类型 | 典型表现 | 风险等级 | 检查方法 |
|---------|---------|---------|---------|
| 安全漏洞 | SQL 注入、XSS（跨站脚本攻击，一种 Web 安全漏洞）、硬编码密钥 | 高 | 安全扫描 + 人工审查 |
| 逻辑错误 | 边界条件未处理、空指针 | 高 | 写测试 + 手动走一遍逻辑 |
| 性能问题 | N+1（一种数据库查询性能问题，循环中逐条查询而非批量查询）查询、内存泄漏 | 中 | 跑基准测试 + 看 profiling（性能分析，找出代码的耗时瓶颈） |
| 代码风格 | 命名不一致、缺少错误处理 | 低 | Lint 工具 + 代码审查 |
| 幻觉代码 | 调用不存在的 API、伪造函数 | 高 | 实际运行 + 查文档验证 |
最后一类"幻觉代码"特别值得注意。AI 有时候会非常自信地调用一个根本不存在的函数或库。代码看起来完全合理，但你一跑就报错。
### 三遍审查法
我用的审查流程，简单但有效：
### 先跑起来
**第一遍：跑起来看效果。**
不要先读代码。先运行，看结果是不是你想要的。如果运行结果就不对，读代码也没意义——直接让 AI 重写。
### 扫关键路径
**第二遍：扫关键路径。**
不用逐行读。只看三个地方：
1. 输入处理 — 参数验证、类型检查做了没有
2. 核心逻辑 — 主流程的思路对不对
3. 输出处理 — 返回值、错误处理、边界情况
### 查安全依赖
**第三遍：查安全和依赖。**
1. 有没有硬编码的密钥、Token、密码
2. 有没有 SQL 拼接、eval（一个危险的 Python 函数，能执行任意字符串代码）、未转义的用户输入
3. 引入的新依赖是否必要、是否可信
三遍做完，大概 5-10 分钟。比你从零写快得多，但足以挡住大部分问题。
![三遍审查法](images/three-pass-review.png)
### 让 AI 帮你审查 AI
一个有用的技巧：让 AI 审查自己（或其他 AI）生成的代码。
在 Cursor 或 Claude Code 里：

```text
请审查以下代码，重点关注：
1. 安全漏洞
2. 逻辑错误
3. 性能问题
4. 是否有更好的实现方式
[粘贴代码]
```

AI 审查 AI 不是完美的，但它能帮你发现很多你一眼看不到的问题。两个不完美的审查者加在一起，比一个强。
### 自动化质量门禁
除了人工审查，用工具设几道自动门禁：
| 门禁 | 工具 | 作用 |
|------|------|------|
| Lint | ESLint / Ruff | 代码风格问题 |
| 类型检查 | TypeScript / mypy | 类型错误 |
| 单元测试 | pytest / Jest | 逻辑正确性 |
| 安全扫描 | Snyk / Bandit | 已知漏洞 |
| Git Hook | pre-commit | 提交前自动检查 |
在项目一开始就让 AI 帮你配好这些工具。之后每次 AI 生成代码，门禁自动帮你挡住一部分问题。
> **核心建议**
>
> 审查 AI 代码不是不信任 AI，是对用户负责。AI 不在乎代码跑在生产环境还是本地测试环境。你在乎。所以审查是你的责任，不是可选项。花 5 分钟审查，省 5 小时排故障。

## §07 走偏了就调头：迭代循环与重构策略
When GPS Leads You Astray: Iteration Loops and Refactoring
你不会第一次跟 AI 说需求就得到完美的代码。这不正常吗？太正常了。
你跟 GPS 说"去机场"，它给你规划了一条路线。你开了一会儿发现前方修路，怎么办？你不会扔掉 GPS 徒步走，你会让它重新规划路线。
Vibe Coding 的迭代也是这个道理。
### 三种迭代模式
根据问题的性质，有三种不同的迭代策略：
### 微调
**模式一：微调。** 方向对，细节要改。
AI 生成的代码 80% 没问题，但有几个小地方需要调整。比如变量命名、错误提示文案、某个参数的默认值。
策略：直接告诉 AI 改哪里。"把第 15 行的超时时间从 5 秒改成 10 秒"、"错误信息改成中文"。
### 转向
**模式二：转向。** 方向不对，重新来。
AI 理解错了你的需求，或者选了一个你不认同的实现方式。代码能跑，但架构不是你想要的。
策略：别在错误的方案上修补。回到起点，重新描述需求，加上这次你学到的东西。"不要用类的方式写，改成函数式的"、"不需要 ORM，直接写 SQL"。
### 拆解
**模式三：拆解。** 需求太大，AI 吃不下。
一次提的需求太多，AI 生成的代码虽然能跑但很混乱。
策略：把一个大任务拆成几个小任务，一步步来。先让 AI 写核心逻辑，确认没问题后再加错误处理，再加测试，再加文档。
![三种迭代模式](images/iteration-modes.png)
### 迭代的节奏控制
什么时候该微调，什么时候该推倒重来？有个简单的判断标准：
| 状况 | 策略 | 为什么 |
|------|------|--------|
| 改了两轮还没对 | 转向 | 方向可能就错了，继续修补是浪费时间 |
| 代码越改越乱 | 拆解 | 任务可能太大，拆小了更容易控制 |
| 只差最后一公里 | 微调 | 都走到这了，别轻易放弃 |
| AI 反复犯同一个错 | 换工具/换模型 | 当前模型可能不适合这个任务 |
一个重要的经验：**不要在一个错误的方向上迭代超过三轮。** 三轮还没改好，说明问题不在细节，在方向。重新来。
### Git 是你的安全网
每次 AI 生成一段你能接受的代码，立即提交。这样如果后面的修改搞砸了，你可以轻松回退。

```bash
# AI 帮你完成了用户注册功能，测试通过
git add -A && git commit -m "feat: add user registration"
# AI 继续做登录功能，但搞乱了注册功能
git diff  # 看看改了什么
git checkout -- routes/auth.py  # 只回退被搞乱的文件
```

这个工作流有一个名字：**小步快跑，频繁提交。** 每完成一个小功能就提交，永远有一个可以回退的安全点。
### 重构：什么时候该让人来
AI 擅长写新代码，不太擅长理解复杂的历史代码。当你的项目越来越大，你会发现 AI 开始"乱改"——改了一个地方，坏了另一个地方。
这时候需要人来主导重构：
1. 你来规划重构方案（哪些模块要动、怎么解耦）
2. 你来决定重构的顺序（先改接口还是先改实现）
3. 让 AI 按你的方案逐个模块重构
4. 每重构一个模块就跑测试
> **核心建议**
>
> 迭代是 Vibe Coding 的常态，不是异常。不要追求一次到位。第一次生成"能用"，第二次迭代"好用"，第三次打磨"漂亮"。三步走比一步到位快得多。GPS 给你规划路线也不是一次就最优的——它会根据路况实时调整。

## §09 第一次上路：用自然语言写一个 CLI 工具
Your First Drive: Building a CLI Tool with Natural Language
到了最激动人心的部分：用 Vibe Coding 从零写一个能用的工具。
我们要做一个文件整理器。功能很简单：扫描指定目录，按照文件类型（图片、文档、视频、音频、其他）自动分类到不同子文件夹。
没有 AI，这个工具大概需要 1-2 小时。用 Vibe Coding，10-15 分钟。
### 第一步：描述需求
在 Cursor 里打开一个新的对话窗口（Cmd+L），输入：

```text
你是一个 Python 开发者。
帮我写一个命令行工具，功能是整理指定目录下的文件。
要求：
1. 扫描用户指定的目录
2. 按文件扩展名分类到子文件夹：images（jpg/png/gif/webp）、documents（pdf/docx/txt/xlsx）、videos（mp4/mov/avi）、audio（mp3/wav/flac）、others（其余所有）
3. 用移动操作（不是复制）
4. 显示处理了多少个文件
5. 支持 --dry-run 模式（只显示会做什么，不实际移动）
用 Click 框架，Python 3.12+。
```

AI 会生成一个完整的 Python 文件。我们来看看它大概会给你什么。
### 第二步：审查和运行
AI 生成的第一版代码，先别读。直接跑。

```bash
# 先看看帮助信息
python organizer.py --help
# 用 dry-run 模式测试一下（不会真的移动文件）
python organizer.py ~/Downloads --dry-run
```

如果 dry-run 的输出符合预期——文件被正确分类了——那核心逻辑没问题。接下来正式运行：

```bash
python organizer.py ~/Downloads
```

### 第三步：发现问题时迭代
大概率第一版不会完美。常见的问题：
**问题 1：没有处理子目录。** 你可能只想整理当前目录的文件，不想递归处理子目录。或者你想递归处理。
解决方案：告诉 AI 你的需求。

```text
文件整理工具需要支持递归扫描子目录。
加一个 --recursive 参数，默认不递归。
```

**问题 2：目标文件夹已经存在同名文件。** 直接移动会覆盖。

```text
移动文件时，如果目标文件夹已有同名文件，自动重命名。
比如 photo.jpg → photo_1.jpg → photo_2.jpg。
```

**问题 3：没有进度显示。** 文件多的时候，黑屏等半天不知道在干嘛。

```text
加一个进度条，显示当前处理进度。
用 rich 库的进度条功能。
```

每改完一个就提交一次：

```bash
git add -A && git commit -m "feat: add recursive scan support"
git add -A && git commit -m "fix: handle duplicate filenames"
git add -A && git commit -m "feat: add progress bar"
```

### 完整的最终版本
经过几轮迭代，你大概会得到这样一个核心结构：

```python
@click.command()
@click.argument("directory", type=click.Path(exists=True))
@click.option("--dry-run", is_flag=True, help="只显示操作，不实际执行")
@click.option("--recursive", is_flag=True, help="递归处理子目录")
def organize(directory: str, dry_run: bool, recursive: bool):
    """整理指定目录下的文件，按类型分类到子文件夹。"""
    # 用 Path 而不是字符串拼接，避免跨平台路径问题
    target = Path(directory)
    # glob 的 ** 表示递归，rglob 会自动递归子目录
    pattern = "**/*" if recursive else "*"
    files = [f for f in target.glob(pattern) if f.is_file()]
    # 如果文件太少就不需要进度条了
    if len(files) > 20:
        progress = Progress()
        with progress:
            task = progress.add_task("整理中...", total=len(files))
            for f in files:
                move_file(f, target, dry_run)
                progress.advance(task)
    else:
        for f in files:
            move_file(f, target, dry_run)
```

这不是伪代码，是真正能跑的代码。从描述需求到得到这个版本，如果你跟着做，应该不超过 20 分钟。
> **核心建议**
>
> 第一个项目的关键是"跑起来"，不是"写完美"。10 分钟出第一版 → 测试 → 发现问题 → 5 分钟改好 → 再测试 → 再改。三轮迭代，比你在编辑器里想 2 小时然后一次写完快得多。

## §10 导航出错时：调试和排错实战
When Navigation Fails: Debugging and Troubleshooting in Practice
AI 写的代码能一次跑通吗？有时候能。大多数时候不能。
别紧张，这不是 AI 的问题。你自己写的代码第一次就能跑通吗？
调试是编程的一部分。只不过 Vibe Coding 的调试方式不太一样。
### 传统调试 vs Vibe Coding 调试
| 维度 | 传统调试 | Vibe Coding 调试 |
|------|---------|-----------------|
| 看报错 | 你自己看 | 把报错贴给 AI |
| 定位问题 | print / debugger | 问 AI "这个报错是什么意思" |
| 修复方式 | 你自己改代码 | 告诉 AI 报错信息，让它改 |
| 验证 | 你自己跑测试 | AI 跑测试，你看结果 |
| 耗时 | 10-60 分钟 | 1-10 分钟 |
效率差异巨大。但前提是你得会"问"。
### 把报错信息喂给 AI 的正确姿势
差的做法：只截一小段报错。

```text
报错了：TypeError: cannot unpack non-iterable NoneType object
```

AI 会猜。猜大概率是猜错的。
好的做法：报错信息 + 上下文 + 你在做什么。

```text
运行 python organizer.py ~/Downloads 时报错。
完整报错：
TypeError: cannot unpack non-iterable NoneType object
  File "organizer.py", line 42, in categorize_file
    category, ext = FILE_TYPES[ext]
我在上一个版本基础上加了自定义分类功能。
请分析原因并修复。
```

### 四步调试法
遇到 AI 的代码报错，按这个顺序处理：
### 贴报错
**第一步：把完整报错贴给 AI。** 包括 traceback、文件名、行号。不要截断。
### 给上下文
**第二步：告诉 AI 你之前做了什么。** "我刚加了递归扫描功能"或者"我把 Click 升级到了 8.2"。上下文很重要。
### 要解释
**第三步：让 AI 解释原因，不只是修代码。** "先告诉我为什么错了，再给我修复方案。" 这一步帮你理解问题，而不只是换个能跑的代码。
### 加防护
**第四步：让 AI 加防御性代码。** "修复这个问题，并在类似的地方加上防护。" 防止同一个坑踩两次。
### 常见报错模式
AI 生成代码中有几类特别常见的错误：
| 报错类型 | 根因 | 预防措施 |
|---------|------|---------|
| ImportError / ModuleNotFoundError | AI 用了你没装的库 | 在 Prompt 里列出已安装的包 |
| AttributeError | AI 调用了不存在的方法（幻觉） | 让 AI 确认 API 是否存在 |
| TypeError | 类型假设错误 | 要求 AI 加 type hints |
| FileNotFoundError | 路径硬编码了绝对路径 | 要求 AI 用相对路径 + 配置 |
| IndentationError | AI 生成的代码缩进混乱 | 用 Lint 工具自动格式化 |
### 复杂 Bug 的策略
有些 bug 不是一眼能看出来的。这时候可以用"分治法"：

```text
这段代码的输出不符合预期。
请帮我逐步调试：
1. 先在每个关键步骤后加 print，输出中间结果
2. 我会运行并把输出贴给你
3. 你根据中间结果判断问题在哪
```

这比直接让 AI 猜"为什么不对"有效得多。因为你给了 AI 实际的数据，而不是让它凭空推理。
> **核心建议**
>
> 调试 Vibe Coding 项目时，你的角色从"修代码的人"变成了"描述问题的人"。你不需要知道怎么修，但你必须能准确描述"哪里不对"。这就像跟修理厂描述车的问题——你不需要会修发动机，但你要能说清"加速时有异响"。

## §11 从车库到马路：测试、部署与上线
From the Garage to the Road: Testing, Deployment, and Going Live
你的代码在本地跑通了。恭喜。但那只是停在车库里。
要让代码真正有用，你需要测试保证它靠谱，部署让它能被访问。
### 让 AI 写测试
测试是 Vibe Coding 最被低估的用途。让 AI 写功能代码容易出错，但让 AI 写测试非常靠谱——因为测试的逻辑通常比业务逻辑简单、明确。

```text
为文件整理器 organizer.py 写单元测试。
覆盖以下场景：
1. 正常分类各种文件类型
2. 重复文件名自动重命名
3. dry-run 模式不实际移动文件
4. 空目录不报错
5. 不存在的目录抛出明确异常
用 pytest，测试文件放在 tests/ 目录。
```

AI 生成的测试大概率一次就能跑通。因为测试的需求描述通常比业务需求更精确。

```bash
# 运行测试
pytest tests/ -v
# 看覆盖率
pytest tests/ --cov=src --cov-report=term-missing
```

### CI/CD：让测试自动跑
如果你把代码放在 GitHub 上，可以让 AI 帮你配一个 CI 流水线：

```text
帮我创建一个 GitHub Actions 工作流：
1. 每次 push 和 PR 都自动跑 pytest
2. Python 3.12 环境
3. 先安装依赖再跑测试
4. 测试失败时阻止合并
```

AI 会给你一个 `.github/workflows/test.yml` 文件。放到项目里就行。以后每次你提交代码，测试自动跑。AI 生成的代码有问题？CI 会帮你挡住。
### 部署的选择
部署方式取决于你的项目类型：
| 项目类型 | 推荐部署方式 | 适合场景 |
|---------|-------------|---------|
| CLI 工具 | PyPI / Homebrew | 给其他开发者用 |
| Web API | Railway / Fly.io | 需要在线访问的服务 |
| 静态网站 | Vercel / Netlify | 前端项目 |
| 全栈应用 | Render / Railway | 前后端一体的项目 |
| Docker | 任何云平台 | 需要环境一致性的项目 |
当然，部署也可以让 AI 帮你。比如：

```text
帮我创建一个 Dockerfile：
1. 基于 python:3.12-slim
2. 安装依赖
3. 设置入口点为 organizer.py
4. 暴露健康检查端口
```

### 发布前的检查清单
代码能跑 ≠ 可以发布。上线前过一遍这个清单：
- [ ] 所有测试通过（pytest / Jest）
- [ ] 没有 print 调试语句残留
- [ ] 没有 hardcoded 密钥或 Token
- [ ] .env 文件在 .gitignore 里
- [ ] README 写了安装和使用方法
- [ ] 依赖版本在 requirements.txt / package.json 里锁定
- [ ] 加了 LICENSE 文件
> **核心建议**
>
> 测试是 Vibe Coding 里投入产出比最高的事情。让 AI 写功能代码你可能要改三遍，但让 AI 写测试几乎一次到位。先让 AI 写测试，再让 AI 写功能代码通过测试——这个"测试驱动"的 Vibe Coding 流程，比"先写代码后补测试"靠谱得多。

## §13 实战二：自动化工作流脚本
Project 2: Automating Your Workflow
日常开发里有很多重复性操作：每次部署要跑一堆命令、每次发版要改好几个文件、每周要汇总数据做报告。
这些事情不难，但烦。Vibe Coding 最擅长解决的就是这类"重复性但不是核心业务"的工作。
### 场景：自动周报生成器
假设你是一个开发者，每周五需要写一份周报，内容包括：
- 本周完成了哪些 commit
- 哪些 PR 合并了
- 代码统计（新增/删除/修改行数）
- 下周计划（需要手动填写）
### 第一步：描述需求

```text
帮我写一个 Python 脚本，自动生成本周工作周报。
输入：Git 仓库路径
输出：Markdown 格式的周报
需要：
1. 用 git log 获取本周（周一到今天）的 commit 记录
2. 用 git log --stat 统计代码变更行数
3. 用 git shortlog 按日期分组 commit
4. 生成 Markdown 格式的周报模板，包含以上信息
5. 预留"下周计划"部分供手动填写
用 subprocess 调用 git 命令，不需要 GitPython 库。
```

### 第二步：AI 生成的核心代码

```python
def get_week_commits(repo_path: str) -> list[dict]:
    """获取本周的 commit 记录。"""
    # 用 --since 限定时间范围，比在 Python 里过滤高效
    monday = get_last_monday()
    result = subprocess.run(
        ["git", "log", "--since", monday, "--pretty=format:%H|%s|%ai"],
        cwd=repo_path,
        capture_output=True,
        text=True,
    )
    commits = []
    for line in result.stdout.strip().split("\n"):
        if not line:
            continue
        hash_val, subject, date = line.split("|", 2)
        commits.append({"hash": hash_val[:8], "subject": subject, "date": date})
    return commits
```


```text
加一个 --team 参数，支持汇总多个项目的工作：

2. 汇总每个仓库的 commit
3. 按项目分组展示
```

```text
加一个功能：自动检测 git remote 是 GitHub 的话，
获取本周合并的 PR 列表（用 gh 命令）。
```

## §14 实战三：给现有项目加新功能
Project 3: Adding Features to an Existing Project
前面的项目都是从零开始。但实际工作中，更常见的场景是：你有一个已有的项目，需要加新功能。
这对 Vibe Coding 是更大的挑战。因为 AI 需要理解已有的代码结构和逻辑。
### 场景：给一个 Flask 博客加评论功能
假设你有一个用 Flask 写的博客系统，现在要加评论功能。
### 第一步：让 AI 理解项目
不要上来就说"加评论功能"。先让 AI 熟悉项目。

```text
请先阅读以下文件，理解项目结构：
- app.py（主入口）
- models.py（数据库模型）
- routes/（路由目录）
- templates/（模板目录）
阅读完成后，告诉我：
1. 这个项目的技术栈和结构
2. 现有的数据库模型
3. 路由的组织方式
```

AI 会读你的代码然后给你一个总结。你看它的理解是否正确。如果有偏差，补充信息。
### 第二步：设计新功能

```text
在理解项目的基础上，帮我设计评论功能：
1. 评论关联到文章（Post 模型）
2. 支持匿名评论（名字 + 内容，不需要登录）
3. 评论按时间倒序显示
4. 简单的防垃圾：同一 IP 每分钟最多评论一次
请先给出数据库模型设计，不要写代码。
```

先要设计，再写代码。这是 Vibe Coding 处理已有项目的重要原则。
### 第三步：逐模块实现
设计确认后，按模块让 AI 写代码：

```text
先写 Comment 模型和数据库迁移。
```

```python
class Comment(Base):
    __tablename__ = "comments"
    id = Column(Integer, primary_key=True)
    # 关联到 Post，用外键保证引用完整性
    post_id = Column(Integer, ForeignKey("posts.id"), nullable=False, index=True)
    author_name = Column(String(100), nullable=False)
    content = Column(Text, nullable=False)
    # 存 IP 用于防垃圾评论，不展示给用户
    ip_address = Column(String(45), nullable=False)
    created_at = Column(DateTime, default=lambda: datetime.now(timezone.utc))
    # 建立和 Post 的关系，方便 post.comments 直接访问
    post = relationship("Post", backref="comments")
```

然后逐步加路由、前端、测试。每一步都确认后再继续。
### 已有项目的注意事项
给现有项目加功能比从零开始更考验你的把控能力：
| 风险 | 表现 | 对策 |
|------|------|------|
| AI 不理解现有架构 | 生成的代码风格和项目不一致 | 先让 AI 读代码，确认它理解了再写 |
| 改错了现有功能 | 加评论时把文章展示搞坏了 | 每次改动后跑全部测试 |
| 代码冲突 | AI 改了别人正在改的文件 | 用 Git 分支，不在 main 上直接改 |
| 上下文溢出 | 项目太大，AI 看不完 | 只给 AI 看相关的文件，别喂整个项目 |
### 关键原则：增量式推进

```text
正确的顺序：
1. 让 AI 读懂项目 → 确认理解正确
2. 让 AI 设计方案 → 你审查方案
3. 让 AI 写模型 → 你审查模型
4. 让 AI 写路由 → 你审查路由
5. 让 AI 写前端 → 你审查前端
6. 让 AI 写测试 → 你跑测试
每一步确认后才开始下一步。
```

不要一口气让 AI 全做完。已有项目的改动牵一发而动全身，增量式推进才能保证每一步不出问题。
> **核心建议**
>
> 在已有项目里用 Vibe Coding，关键是控制 AI 的"视野"。不要把整个项目都喂给它，只给它看和当前任务相关的文件。就像你让导航带你去一个地方，不需要把整张地图都展开——你只需要看到当前位置到目的地之间的路线。

### 什么时候该换工具
工具迭代很快。今天的最佳选择可能半年后就不是了。几个信号告诉你该重新评估了：
1. 你的 AI 工具频繁生成错误代码，且迭代多轮修不好
2. 你花在等 AI 响应上的时间比写代码还多
3. 团队里其他人用的工具明显比你高效
4. 有新工具发布了并且有可靠的评测数据
> **核心建议**
>
> 别纠结"哪个最好"。挑一个主力工具（Cursor 或 Claude Code），深度用两周。用顺了再加第二个工具处理主力不擅长的场景。工具组合 > 单一工具。就像 GPS 导航：日常用 Google Maps，去野外用 Gaia GPS。不是一个地图打天下。

## §16 Vibe Coding 之后：职业、教育与软件工程的未来
After Vibe Coding: Careers, Education, and the Future of Software Engineering
最后一章，不写代码了。聊聊大的。
### 一个不舒服的事实
Vibe Coding 不会取代程序员。但会写 Vibe Coding 的程序员，会取代不会写的程序员。
这句话不是危言耸听。你回想一下 GPS 出现后出租车行业发生了什么：司机没被 GPS 取代，但不会用导航的司机被会用导航的司机取代了。GPS 改变的是竞争门槛，不是岗位本身。
Vibe Coding 在做同样的事。
### 编程能力的新定义
2024 年之前，"会编程"的意思是：能手写代码实现功能。
2026 年之后，"会编程"的意思变成了：
| 维度 | 传统定义 | 新定义 |
|------|---------|--------|
| 核心能力 | 写代码 | 描述需求 + 审查代码 |
| 效率标准 | 一天写多少行 | 一天交付多少功能 |
| 价值来源 | 实现细节 | 架构决策和业务理解 |
| 学习重点 | 语言语法和 API | 系统思维和沟通表达 |
| 竞争优势 | 代码写得快 | 需求想得清楚 |
这不是说编程技术不重要了。而是说，"写代码"从核心能力变成了基础能力。就像开车：会踩油门和刹车是基础，但导航和路线规划才是决定你能不能准时到的关键。
### 不会被 AI 取代的能力
有些能力是 AI 短期内无法替代的：
| 能力 | 为什么 AI 做不了 |
|------|-----------------|
| 理解模糊的业务需求 | 需要和人沟通、理解上下文、做判断 |
| 系统架构设计 | 需要权衡利弊、考虑长期演进 |
| 代码审查和决策 | 需要对质量标准和风险有判断力 |
| 调试复杂问题 | 需要"诊断思维"，不是模式匹配 |
| 团队协作 | 需要沟通、妥协、领导力 |
| 创造性解决问题 | 需要跨领域联想和创新思维 |
### 新手起步
**学生 / 刚入行（0-2 年）：**
一定先学编程基础。Python、Git、数据库、HTTP——这些是基本功。没有驾驶技术，GPS 再好也白搭。
然后用 Vibe Coding 加速学习。遇到不懂的概念，让 AI 给你解释、写示例代码。学习效率能提升 3-5 倍。
但别跳过"手写代码"的阶段。你必须经历"一行一行写、一个 bug 一个 bug 调"的过程，才能真正理解代码是怎么工作的。
### 中级进阶
**中级开发者（2-5 年）：**
这是 Vibe Coding 最大的受益群体。你已经知道什么是好代码，只是写起来费时间。AI 帮你省时间，你帮 AI 把关质量。
重点学：Prompt 工程、代码审查技巧、AI 工具的高级用法。把你的经验通过 CLAUDE.md 和自定义规则传递给 AI，让它按你的标准写代码。
### 资深提效
**资深开发者（5 年以上）：**
你最大的挑战不是写代码，是做决策。技术选型、架构设计、团队协作——这些 AI 帮不了太多。
但 Vibe Coding 可以帮你快速验证想法。以前你想试一个架构方案要花一天搭 prototype，现在 30 分钟就能看到效果。

### 最后的建议
回到 GPS 隐喻。
GPS 改变了出行方式，但它没有改变出行需要人这件事。你仍然需要决定去哪、判断路况、处理突发情况。
Vibe Coding 改变了编程方式，但它没有改变编程需要人这件事。你仍然需要理解需求、设计架构、把控质量。
唯一的问题是：你是愿意当那个会用 GPS 的司机，还是坚持凭记忆认路的人。

>
> 不要等"AI 编程成熟了"再开始学。现在就是最好的时候。工具已经够用，社区资源已经丰富，唯一缺的是你的实践。跟着这本教程做三个项目，你就会超越 90% 还在观望的人。GPS 不是等完全精准了才用的——它从第一天起就比凭记忆认路强。
